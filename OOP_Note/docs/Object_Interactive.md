# 构造、清理与对象交互

第四节系统阐述了 C++ 的构造与清理机制（C'tor 与 D'tor）、对象初始化列表的底层细节、函数重载与参数匹配规则、常量对象与常成员函数、以及 C++ 内存管理责任归属与参数传递的最佳实践。

---

### 构造函数与析构函数

C++ 引入构造函数与析构函数，从语言层面保证了对象生命周期内“安全初始化”和“安全清理”的职责。

#### 构造函数（Constructor）
* **引入原因**：手动编写初始化函数（如 `init()`）极易被程序员遗忘，从而导致程序中存在未初始化的脏对象。
* **规则**：
    * 函数名与类名完全相同。
    * **无返回值**（连 `void` 都没有）。
    * 编译器保证在对象创建的时刻，**自动调用**构造函数，在用户真正使用该对象前完成初始化。
* **参数与重载**：构造函数可以拥有参数（如 `Tree(int i)`），此时创建对象必须显式传参（如 `Tree t(12);`）。

#### 默认构造函数（Default Constructor）
* **定义**：可以被**无参调用**的构造函数（包含参数列表全部有默认值的情况）。
* **对象数组的创建与默认构造**：
    * 声明对象数组时（如 `Y y[5]`），系统会对数组中的每个元素调用默认构造函数。
    * 如果类定义了带参构造函数，但**没有定义默认构造函数**，那么<u>创建对象数组时必须显式全部初始化</u>（如 `Y y[] = { Y(1), Y(2) }`）。任何导致隐式调用无参构造的行为都会触发编译错误。
* **合成的默认构造函数（Auto Default Constructor）**：当且仅当一个类**没有定义任何构造函数**时，编译器才会自动合成一个无参的默认构造函数。一旦手动定义了任何构造函数，编译器就不会再自动合成。

#### 析构函数（Destructor）
* **规则**：名称为类名前加波浪号（`~Class()`），**无参数且无返回值**。
* **调用时机**：由编译器在对象生命周期结束时自动调用：
    * 栈对象：超出其所在的作用域大括号 `{}`。
    * 堆对象：执行 `delete` 或 `delete[]` 释放内存时。
* **职责**：释放动态分配的内存资源（`new` 出来的空间）、关闭文件描述符或数据库连接等。

!!! danger "重要规则：不可跨越变量初始化跳转（No-jump 规则）"
    C++ 编译器必须严格保证任何对象的构造和析构函数被成对、顺序调用。因此，**禁止使用 `goto` 或 `switch-case` 直接跳转并越过变量的初始化语句**（跨越其定义点直接进入其作用域），否则会导致编译错误：
    `jump to case label crosses initialization of 'int x'`
    *注：对于 `int` 等基本类型，C++ 允许通过类似构造函数的语法进行初始化（如 `int a(10)` 或 `int b = int(20)`）。*

### 对象交互与作用域遮蔽

#### 变量分类与生命周期对比
* **成员变量（Fields）**：定义在类内部、成员函数之外。生命周期与对象一致，具有**类作用域**，类内的所有成员函数均可访问。
* **局部变量与形参**：定义在函数内部或参数列表中。生命周期极短，随着单次函数调用的执行而创建、调用完毕即销毁。具有**局部/块作用域**。

#### 作用域遮蔽（Shadowing）与 `this` 解决
如果局部变量或参数与成员变量同名，会导致在函数内部无法直接访问该成员变量：

??? example "点击查看代码"
    ```cpp
    int TicketMachine::refundBalance() {
        int balance; // 同名局部变量遮蔽了成员变量 balance
        balance = 0; // 此时修改的是局部变量，成员变量未受影响！
    
        // 解决方法：利用 this 指针显式指明
        this->balance = 0; 
    }
    ```

### 初始化列表（Initializer List）

#### 初始化列表语法

??? example "点击查看代码"
    ```cpp
    class Point {
    public:
        Point(float xa = 0.0, float ya = 0.0) : x(xa), y(ya) {} // 初始化列表
    private:
        const float x, y;
    };
    ```

* **运行机制**：初始化列表中的初始化动作发生在**进入构造函数体 `{}` 之前**。
* <u>**强制使用场景**</u>：
    * **`const` 成员变量**：常量必须在定义时初始化，进入 `{}` 后只能进行“赋值”操作，因此常量成员必须在初始化列表中初始化。
    * **引用成员变量**：引用在声明时必须绑定，因此同样必须使用初始化列表。
    * **无默认构造函数的成员对象**：若成员对象没有默认构造函数，必须在列表中显式调用其有参构造函数。

#### 关键细节：初始化顺序
* **初始化的真实顺序**是**成员变量在类定义中的声明顺序**，而**不是**它们在初始化列表中的排列顺序！

    举个例子：
    ??? example "点击查看代码"
        ```cpp
        class Trap {
        private:
        int y; // 声明顺序：y 在前
        int x; // 声明顺序：x 在后
        public:
        // 直觉上：先给 x 赋值 10，再用 x 的值加 5 去初始化 y，此时 y 应该等于 15 吧？
        // 实际上：错！因为声明中 y 在前，编译器会先执行 y(x + 5)。但此时 x 还没被创建（是个未初始化的垃圾值），所以 y 最终会得到一个随机的垃圾数据！之后编译器才执行 x(10)。
        Trap() : x(10), y(x + 5) {} 
        };
        ```

* <u>**析构顺序**</u>则与初始化顺序完全相反（后构造的先析构）。
* *注：C++11 允许在声明成员变量时直接赋予初值（如 `int i = 0;`），这会被所有未显式覆盖该值的构造函数采用。*

### 函数重载与默认参数

#### 函数重载（Function Overloading）
* **定义**：同一作用域内允许多个同名函数共存，但它们的**参数列表（类型、个数或顺序）必须不同**。**返回值类型不同不能作为重载的依据**。
* **自动类型匹配与二义性错误**：
  当调用重载函数传入的实参类型与定义不完全一致时，编译器会尝试进行**隐式类型转换/提升**。如果编译器发现多种转换路径都可行且“平分秋色”（如 `f(double)` 和 `f(short)` 都可以匹配 `f(2)`），将抛出**二义性匹配错误（Ambiguous call）**，拒绝编译。

#### 默认参数（Default Arguments）
* **定义**：在函数声明中为参数指定默认值，调用时若省略该实参，编译器将自动插入默认值。
* **语法限制**：默认参数的指定必须**从右向左**连续进行，中间不能有间隔。

    ??? example "点击查看代码"
        ```cpp
        int harpo(int n, int m = 4, int j = 5); // 合法
        int chico(int n, int m = 6, int j);     // 非法！默认参数右侧存在无默认值的参数
        ```

### 常量对象与常成员函数

#### 常量对象（Const Objects）
使用 `const` 声明的对象（如 `const Date birthday(12, 25, 1994);`），一旦完成初始化，其所有数据成员均变为“只读”，不可被修改。

#### 常成员函数（Const Member Functions）
为了在不修改常量对象的前提下访问其数据，必须使用常成员函数。

* **语法**：在函数声明与实现的参数列表后均须附带 `const` 关键字。

    ??? example "点击查看代码"
        ```cpp
        int get_day() const; // 声明
        int get_day() const { return day; } // 定义
        ```

* **行为限制**：
    * 常成员函数**禁止修改**对象的任何非静态数据成员。
    * 常成员函数**禁止调用**任何非 `const` 成员函数。
    * **常量对象只能调用常成员函数**；普通（非 const）对象既可以调用普通成员函数，也可以调用常成员函数。

#### 类内部的编译时常量
如果想在类定义内部定义一个常量来作为数组成员的长度，直接声明 `const int size;` 会报错，因为每个对象的 `size` 可以在初始化列表中不同，不是编译期确定的。

错误原因: 普通数组的大小必须在编译阶段就确定。虽然 size 是 const，但它是“对象级”的常量。每个不同的对象实例，其 size 可以在初始化列表中被设为不同的值。编译器在编译类定义时，并不知道具体的 size 是多少，所以不能用它定义数组。

* **解决方法一（推荐）**：使用 `static const int size = 100;`。加上 `static` 后，该常量属于类级共享，编译器可以在编译期直接确定其数值。
* **解决方法二（旧版 Hack 技巧）**：使用匿名枚举（Enum Hack）。

    ??? example "点击查看代码"
        ```cpp
        class HasArray {
            enum { size = 100 }; // 枚举在编译时计算且不占用对象物理空间
            int array[size];     // 合法
        };
        ```

### 参数传递与内存管理职责

#### 传递与返回设计指南（TIPS）
1. **入参设计**：
    * 如果函数仅仅是为了**读取**传入对象的值，应当传入 **`const` 引用**（如 `void f(const Student& s)`）或 **`const` 指针**。这既避免了深拷贝开销，又保证了安全性。
    * 如果函数需要修改传入的对象，传入**普通引用**或**普通指针**。
    * 如果需要将对象存入容器或保存其副本，直接按值传入对象（`void f(Student s)`）。
2. **出参设计**：
    * 如果对象是在函数内部创建的局部变量，**直接按值返回对象**（`Student f()`）。现代 C++ 的返回值优化（RVO）会消除多余的拷贝开销，切忌返回局部变量的指针或引用（会变成野指针/野引用）。
    * 只有在返回的对象生命周期长于函数本身（如传入的参数对象、全局/静态变量、或者类的成员变量）时，才可以返回指针或引用。

#### 内存管理的责任归属原则
* **理想原则**：**“谁申请，谁释放”**。
* **避坑准则**：**永远不要在函数内部 `new` 一个对象并返回其指针**（如 `char *foo() { return new char[10]; }`），这会导致对象的所有权和生命周期极其模糊，使用者极易忘记 `delete`，导致致命的内存泄漏。应当尽量通过返回对象本身（或使用智能指针）来规避此问题。
