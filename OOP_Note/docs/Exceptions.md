# 异常处理机制

本节深入介绍了 C++ 异常处理的哲学、语法结构与运行机制（try/catch/throw 与栈展开）、异常匹配规则、基于 RAII 的异常安全设计、以及构造函数与析构函数中异常处理的特殊铁律。

---

### 异常处理概述

#### 传统错误处理的弊端
在 C 语言等传统开发中，错误处理通常依靠返回特殊的错误码（如 `errorCode`）。这会导致代码中充斥着嵌套的 `if-else` 条件判断（如文件读写示例），使正常业务逻辑与错误恢复逻辑交织在一起，极难维护。

#### 异常的隔离优势
异常机制将“正常执行的逻辑”与“错误处理的逻辑”彻底分离，使得代码结构清晰。

* **抛出（Throw）**：当在某个执行点发现致命错误且当前无法解决时，使用 `throw` 关键字*抛出异常对象，中止当前函数执行。*
* **捕获（Catch）**：调用链上游的函数如果对此错误感兴趣，可以使用 `try-catch` 块来*捕获并处理该异常。*

    ```cpp
    // 示例：抛出自定义异常
    class VectorIndexError {
    public:
        VectorIndexError(int v) : m_badValue(v) {}
        void diagnostic() {
            cerr << "index " << m_badValue << " out of range!" << endl;
        }
    private:
        int m_badValue;
    };
    
    template <class T>
    T& Vector<T>::operator[](int indx) {
        if (indx < 0 || indx >= m_size) {
            throw VectorIndexError(indx); // 抛出异常临时对象
        }
        return m_elements[indx];
    }
    ```

### 异常的传播过程

1. 调用链回溯：当异常被抛出时，控制权会顺着函数的调用栈一层层向上寻找第一个匹配的 `catch` 处理器。
2. 栈展开（Stack Unwinding）：在此回溯过程中，所有因为控制权离开而退出的作用域内的局部栈对象，其析构函数都会被编译器自动调用。这保证了资源的自动回收。
3. 重新抛出（Re-throw）：如果在当前 `catch` 块中只能处理部分错误，或者仅作记录，可以在 `catch` 块内部写一个孤立的 **`throw;`** 关键字，将当前捕获的异常继续原样向上传播。
4. 捕获所有异常：使用省略号 **`catch (...)`** 可以捕获任何类型的异常。通常作为最后一道防线。
5. 未捕获异常：如果一个异常一路上溯到 `main` 之外仍未被任何 `catch` 块捕获，程序将自动调用 **`std::terminate()`** 终止运行。可以通过 `std::set_terminate()` 注册自定义的退出函数。

---

### 异常匹配规则

同一个 `try` 块后面可以跟多个 `catch` 块，编译器在匹配时遵循以下规则：

* 顺序匹配：按照 `catch` 块在代码中出现的**先后顺序**从上到下进行检查，第一个匹配的块将被执行。
* 类型匹配与隐式转换：
    * 匹配要求类型严格一致。
    * **唯一允许的隐式转换是向上转型（Upcasting）**——即基类的引用/指针（`Base&` 或 `Base*`）可以匹配派生类异常。
* 继承体系下的 Catch 顺序：
  由于基类能够匹配派生类，**必须将派生类（子类）的 `catch` 处理器放在基类（父类）的 `catch` 处理器之前**。否则，基类处理器会先拦截并“吞掉”所有子类异常。

??? example "点击查看代码"
    ```cpp
    try {
        throw UnderflowErr();
    } catch (ZeroDivideErr& e) {
        // 捕获特定异常
    } catch (MathErr& e) {
        // 捕获所有继承自 MathErr 的异常（包括 UnderflowErr）
    } catch (...) {
        // 捕获其余所有异常
    }
    ```

### 异常规格说明（Exception Specifications）

* **语法**：在函数原型后跟 `throw(Type1, Type2)` 声明该函数可能抛出的异常类型（例如 `void func() throw(MathErr)`；若不抛出任何异常则声明为 `throw()`）。
* **运行期行为**：这并非编译期检查。如果函数在运行期抛出了未在规格列表中声明的异常并传播出函数，系统将调用 `std::unexpected()`，默认导致程序调用 `std::terminate()` 终止。
* *注：在 C++11 及更高版本中，旧版的 `throw()` 规格说明已被废弃，转而使用更加高效的 `noexcept` 关键字。*

### 异常安全设计与 RAII

#### 不要将异常用于正常控制流
异常抛出伴随着栈展开和运行期查找，开销极大。绝对不要将异常当做普通的控制跳转使用（例如用捕获异常来判断循环结束）。

#### 资源获取即初始化（RAII）
不要在代码中手动编写大量的 `try-catch` 去释放资源。

* 反面教材：在 `try` 内部打开文件，在 `catch` 内部调用 `f.close()`。
* 优秀设计：利用 C++ 对象的生命周期。定义一个 `File` 对象，在**析构函数**中执行 `close()` 动作。如果函数抛出异常，栈展开会自动销毁该 `File` 对象并触发析构函数关闭文件。这种机制称为 **RAII (Resource Acquisition Is Initialization) **。

#### 构造函数中的异常铁律
* 背景：构造函数没有返回值，因此**抛出异常是报告构造失败的唯一优秀方式**。
* 铁律：如果构造函数中抛出了异常，**C++ 认为该对象没有被成功创建，因此其对应的析构函数永远不会被调用！**
* 防漏原则：为了防止内存泄漏，构造函数在抛出异常之前，**必须手动释放其内部在此之前已经成功申请的所有动态资源**（或者将子资源交给智能指针/RAII 成员对象管理，成员对象在被销毁时会自动释放其自身）。

#### 析构函数中的异常铁律

!!! danger "析构函数绝对不能抛出异常！"
    析构函数在两种情况下被调用：1. 正常超出生命周期销毁；2. 发生异常时，进行栈展开自动销毁。
    
    如果在栈展开（已经有一个异常处于激活状态）的过程中，某个对象的析构函数又抛出了一个新异常，C++ 运行时系统将直接调用 **`std::terminate()`** 崩溃退出。
    因此，析构函数内部必须捕获并吞掉所有可能产生的异常，**绝不允许任何异常逃离析构函数**。

### 编程最佳实践：Catch By Reference

在编写 `catch` 语句时，有三种类型接收方式，其优劣对比极为明显：

1. **按值捕获（Catch by value）—— ❌ 避免使用**：
    * 会产生对象的拷贝开销。
    * 会引发**对象切片（Object Slicing）**：如果抛出的是派生类异常 `D`，而用基类 `B` 按值捕获，派生类的特有属性和虚函数多态性将被切除，退化为基类。
2. **按指针捕获（Catch by pointer）—— ❌ 避免使用**：
    * 必须在抛出时使用 `throw new D()`。这导致了严重的耦合：捕获方必须记得手动 `delete` 释放该指针，否则将造成内存泄漏。
3. **按引用捕获（Catch by reference）——  推荐做法**：
    * `catch(Base& b)`：无需进行对象拷贝，效率最高。
    * 完美保留多态性，如果在基类中定义了虚函数 `print()`，通过基类引用调用 `b.print()` 依然可以正确执行子类的重写版本。
    * 无需担心动态内存的释放责任。
