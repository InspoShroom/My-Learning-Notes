# 经典设计模式案例分析：Lexi 文档编辑器

本笔记基于课程课件《Patterns Design-Case Study_改.ppt》进行整理。课件以开发一个所见即所得（WYSIWYG）的文档编辑器 Lexi 为案例，深入剖析了如何解决文档结构设计、算法解耦、跨平台窗口系统兼容、用户命令流控制等 8 个核心设计难题，并依次引入了 8 个经典的 Gang of Four (GoF) 设计模式。

---

### 问题一：文档结构的内部表示（树形层次结构）
* **设计需求**：编辑器需要自由混合文本和图形，并且能够以统一的方式对待单个基本字符、图像以及复杂的物理排版结构（如行 Row、列 Column、表格）。
* **设计困境**：如果不做统一抽象，程序在排版或分析时就必须频繁判断当前节点是字符还是图像，导致代码充斥着分支判断。
* **解决方案**：引入**递归组合（Recursive Composition）**。定义一个通用的图形基类 `Glyph`（字形），其子类既包含原始图形元素（如 `Character`、`Image`），又包含结构化布局元素（如 `Row`、`Column`，它们内部持有一个 `Glyph` 链表）。
* **对应模式：组合模式（Composite Pattern）**
  * **本质**：将对象组合成树形结构以表示“部分-整体”的层次结构。它使得客户对单个对象和复合对象的使用具有一致性。

??? example "点击查看代码"
    ```cpp
    // 统一的字形抽象接口
    public abstract class Glyph {
        public abstract void draw(Window w);
        public abstract Rect getBounds();
        // 树形结构管理
        public abstract void insert(Glyph g, int i);
        public abstract void remove(Glyph g);
        public abstract Glyph child(int i);
    }
    ```

### 问题二：排版格式化算法的解耦
* **设计需求**：编辑器需要将文本折行为行，行折为列，且必须支持不同的排版算法（如快速折行、高质量双向对齐等）。
* **设计困境**：排版算法复杂多变，如果将其直接写在 `Glyph` 类（如 `Row`）内部，当需要添加新的排版策略时，必须修改所有相关的字形类。
* **解决方案**：将排版折行算法独立封装。定义一个抽象类 `Compositor`（排版器），它与 `Composition`（排版内容容器）松耦合。`Composition` 仅负责存储字形，而在需要排版时将其委派给 `Compositor` 的具体子类去计算并插入不可见的换行字形。
* **对应模式：策略模式（Strategy Pattern）**
  * **本质**：定义一系列算法，把它们一个个封装起来，并且使它们可相互替换。本模式使得算法可独立于使用它的客户而变化。

### 问题三：装饰用户界面（边框与滚动条）
* **设计需求**：需要动态为文本编辑区域加上边框（Border）或者滚动条（Scrollbar）。
* **设计困境**：使用继承会引发**类爆炸**（如 `BorderedScrollableTextView`、`BorderedTextView` 等），且在运行期无法灵活撤销或叠加装饰。
* **解决方案**：引入**透明包围（Transparent Enclosure）**。定义一个抽象修饰类 `MonoGlyph` 继承自 `Glyph`，其内部只持有一个子字形指针，并将所有调用转发给子字形。具体的子类 `Border` 在转发 `draw()` 请求的前后执行自己特有的画边框操作。
* **对应模式：装饰模式（Decorator Pattern）**
  * **本质**：动态地给一个对象添加一些额外的职责。就增加功能来说，装饰模式相比生成子类更为灵活。

### 问题四：支持多种视觉外观标准（Look-and-Feel）
* **设计需求**：Lexi 需要兼容多种系统视觉主题（如 Motif、Windows、Mac 控件风格），并支持在启动时甚至运行期动态切换风格。
* **设计困境**：如果在代码中直接实例化具体的控件（如 `new MotifScrollBar()`），会导致代码与特定平台主题高度强耦合。
* **解决方案**：引入工厂机制。定义一个抽象工厂 `GUIFactory`，定义创建各类抽象产品的方法（如 `createScrollBar()`、`createButton()`）。具体的工厂子类（如 `MotifFactory`）负责生产特定风格的具体产品。
* **对应模式：抽象工厂模式（Abstract Factory Pattern）**
  * **本质**：提供一个创建系列相关或相互依赖对象的接口，而无需指定它们具体的类。

---

### 问题五：支持多种不兼容的底层窗口系统
* **设计需求**：Lexi 必须能够跨不同的不兼容底层窗口系统（如 XWindow、Windows API、Mac OS）运行。
* **设计困境**：这不同于产品外观，底层窗口系统的 API 规范完全不同，无法通过一个统一的产品继承树来直接抽象。
* **解决方案**：将逻辑抽象与平台实现彻底分离。定义逻辑窗口层次结构 `Window`（负责 `drawRect`、`drawText` 等高级逻辑）；另外定义一套独立的平台实现层次结构 `WindowImp`（如 `XWindowImp`、`WinWindowImp`，专门封装系统相关的原始绘图调用）。`Window` 类内部持有一个指向 `WindowImp` 的指针，将操作委派给它。
* **对应模式：桥接模式（Bridge Pattern）**
  * **本质**：将抽象部分与它的实现部分分离，使它们都可以独立地变化。

### 问题六：用户操作命令的封装与撤销（Undo/Redo）
* **设计需求**：需要支持新建、保存、剪切、粘贴等多种用户菜单操作，且必须提供统一的、多级撤销（Undo）与重做（Redo）队列。
* **设计困境**：如果将操作直接绑定在菜单项 `MenuItem` 类中，菜单项就必须知晓文档的细节，导致 UI 组件与业务逻辑紧耦合；且无法记录历史命令。
* **解决方案**：将用户请求封装为对象。定义抽象 `Command` 类，包含 `execute()`（执行）和 `unexecute()`（撤销）接口。UI 控件（如 `MenuItem`）在初始化时绑定具体的 `Command` 子类，触发时仅需调用其 `execute()`，并将该命令放入历史队列中以支持多级撤销。
* **对应模式：命令模式（Command Pattern）**
  * **本质**：将一个请求封装为一个对象，从而使你可用不同的请求对客户进行参数化；对请求排队或记录请求日志，以及支持可撤销的操作。

### 问题七：遍历散落在层次结构中的数据
* **设计需求**：拼写检查、公式计算等分析操作需要遍历整个文档树，但文档树内部的不同节点采用了不同的底层容器（如链表、数组、树等）来存储子字形。
* **设计困境**：遍历算法如果不加抽象，调用者就必须写一大堆针对不同容器的遍历逻辑；且无法支持多样化的遍历顺序（如前序、后序、逆序）。
* **解决方案**：引入迭代器抽象。定义 `Iterator` 抽象类，提供标准接口如 `First()`、`Next()`、`IsDone()`、`CurrentItem()`。各个字形子类（如 `Row`）通过工厂方法 `CreateIterator()` 返回针对自己内部数据结构的专属迭代器实例。
* **对应模式：迭代器模式（Iterator Pattern）**
  * **本质**：提供一种方法顺序访问一个聚合对象中各个元素，而又不需暴露该对象的内部表示。

---

### 问题八：无缝扩展拼写检查与排版分析（不破坏字形类）
* **设计需求**：需要对文档执行多种分析动作（如拼写检查、字符统计、语法分析、自动加连字符等），且希望分析动作的添加与字形类解耦（即增加新分析不需要去修改原有的 `Glyph` 类）。
* **设计困境**：如果用普通的遍历配合类型判断（如使用 `instanceof` 或 `dynamic_cast` 检查是否为 `CharacterGlyph`），会导致遍历代码极其臃肿且违反开闭原则。
* **解决方案**：引入**双分派（Double Dispatch）**。定义一个 `Visitor`（访问者）类层次结构，其中包含针对各类具体字形的访问方法（如 `visitCharacter()`、`visitRow()`）。在 `Glyph` 类中定义虚函数 `accept(Visitor& v)`，由子类实现：`v.visitCharacter(this);`（利用动态绑定，将控制权反弹回访问者，自动实现类型匹配）。
* **对应模式：访问者模式（Visitor Pattern）**
  * **本质**：表示一个作用于某对象结构中的各元素的操作。它使你可以在不改变各元素的类的前提下定义作用于这些元素的新操作。

### Lexi 案例设计模式总结清单

| 难题 / 设计目标 | 解决方案 | 采用的设计模式 |
| :--- | :--- | :--- |
| **文档物理结构的内部统一表示** | 树形节点的层次递归嵌套 | **组合模式（Composite）** |
| **排版对齐算法与文档树的解耦** | 将排版逻辑封装在独立的 Compositor 算法类中 | **策略模式（Strategy）** |
| **动态且透明地为编辑区域添加边框/滚动条** | 使用单子节点的透明包围进行能力增强与任务委托 | **装饰模式（Decorator）** |
| **跨系统视觉外观（Look-and-Feel）的一键切换** | 抽象控件创建过程，用 GUIFactory 屏蔽具体类实例化 | **抽象工厂模式（Abstract Factory）** |
| **支持不同的底层不兼容窗口系统开发** | 窗口逻辑层与系统实现层彻底分离，通过指针桥接 | **桥接模式（Bridge）** |
| **支持 UI 操作重用与多级撤销（Undo/Redo）** | 将动作封装为含有 execute/unexecute 的对象 | **命令模式（Command）** |
| **统一遍历由各种不同容器存储的字形树** | 抽象出迭代器接口，屏蔽底层遍历细节 | **迭代器模式（Iterator）** |
| **在不修改字形类的前提下无缝新增拼写检查等分析** | 引入双分派机制，由外部访问者对象累积遍历状态 | **访问者模式（Visitor）** |
