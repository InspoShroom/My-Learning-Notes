    类与封装

    第三节系统介绍了 C++ 中类（Class）的基本定义、封装机制、this 指针的底层本质、头文件与源文件的分离规范（以及头文件防卫），并详细讲解了利用 Makefile 自动化工具进行项目管理与“最小化编译”的原理。

    从结构体到类

    C 语言结构体限制

    在 C 语言中，结构体只能包含数据（Data），而对数据的操作函数则是完全分离的。这种设计缺乏封装性，容易导致外部代码对结构体成员变量的直接误修改。 * 指针参数的 const 保护：在 C 语言中，为了防止函数修改传入的结构体，建议声明为指向常量的指针，如 void print(const Point* p)。如果在函数内尝试修改，编译器将报错。

    C++ 的类与封装

    C++ 将数据（属性 Attributes）和操作数据的函数（服务 Services）捆绑在一起，这就是封装（Encapsulation）。类可以理解为一个更强大的结构体。

    点击查看代码 
    // C++ Point 类的定义
class Point {
public: // 外部接口
    void init(int x, int y);
    void move(int dx, int dy);
    void print() const; // 常成员函数，保证不修改数据成员

private: // 封装的数据，防止外部直接访问
    int x;
    int y;
};

    作用域解析运算符（::）

    • 语法：<类名>::<成员函数名>。用于在外部定义类成员函数，指明该函数属于特定的类作用域。
    • 全局作用域：当 :: 前面不带任何类名时，代表全局作用域（例如 ::f() 调用全局的 f 函数，而不是类内的同名递归）。

    容器与 Stash 案例

    在引入 this 指针之前，我们需要了解 C++ 中容器（Container）的概念，以《Thinking in C++》中经典的 Stash 动态存储容器案例为例。

    1. 容器的概念

    • 定义：容器（Container）是专门用来存放、管理其他对象的对象。
    • 通用接口：对于大多数容器，最核心和通用的操作接口通常是存入数据 put()（或 add()）与取出数据 get()（或 fetch()）。
    • Stash 案例：Stash 是一个教学案例，它是一个可以在程序运行期间动态自动扩容的容器。

    2. Stash 容器的设计理念

    • 按值复制（Clone）：Stash 内部存储的是传入对象的副本（Clone），而不是外部原对象本身。这样可以保证容器内数据的生命周期独立于外部。
    • 无类型容器（Typeless Container）：
      • 它内部通过无类型指针（void* 或 unsigned char* 字节流）来管理内存，不关心存储的具体类型，只关心类型所占用的字节大小（size）。
      • 尽管它是“无类型”的，但在实际使用时，同一个 Stash 实例通常只应存放同一种类型的数据。

    3. C++ 结构体中包含函数

    在 C++ 中，struct 不仅可以包含数据成员，还可以包含成员函数（这与 class 几乎完全一致，唯一的区别在于 struct 默认访问权限为 public，而 class 为 private）。

    以下是 Stash 容器的声明示例：

    点击查看代码 
    // Stash2.h 声明部分
struct Stash {
    int size;      // 每个存储空间的大小（字节数）
    int quantity;  // 当前开辟的存储空间总数
    int next;      // 下一个可用的空闲空间索引
    unsigned char* storage; // 动态分配 of bytes: 字节数组首地址

    // 成员函数！
    void initialize(int size);
    void cleanup();
    int add(const void* element);
    void* fetch(int index);
    int count();
    void inflate(int increase);
};

    4. 成员函数与对象的关联

    当我们声明一个结构体变量并调用它的成员函数时：

    点击查看代码 
    Stash a;
a.initialize(10); // 初始化每个元素大小为 10 字节

    这里，函数 initialize() 必须作用于具体的对象 a。成员函数内部需要知道自己正在对哪个具体的变量进行操作——这就自然地引出了底层的 this 指针机制。

    this 指针：成员函数的隐式参数

    底层本质

    在 C++ 中，每个类的非静态成员函数在编译后，都会被编译器隐式地插入一个参数——名为 this 的指针，其类型为当前类的指针。

    • 函数转换：

      点击查看代码 
      // 逻辑上的代码
void Point::init(int ix, int iy) { x = ix; }

// 编译器翻译后的底层代码
void Point::init(Point* this, int ix, int iy) {
    this->x = ix;
}

    • 调用转换：

      点击查看代码 
      // 逻辑上的代码
Point a;
a.init(10, 20);

// 编译器翻译后的底层代码
Point::init(&a, 10, 20);

      this 指针是所有类成员函数中天然可用的局部变量，指向调用该函数的对象实例。

    面向对象编程（OOP）五大特征

    1. 万物皆为对象：最纯粹的定义是一块存储区域（Region of storage）。
    2. 程序是相互协作的对象群：通过发送消息（调用成员函数）告知彼此该做什么。
    3. 每个对象都有由其他对象构成的内存：支持组合（Composition）。
    4. 每个对象都有一个类型：即所属的类（Class）。
    5. 同类型的所有对象能接受相同的消息：具有接口的一致性。

    独立编译与头文件规范

    声明（Declaration）与定义（Definition）的分离

    为了更好地进行大型项目管理，C++ 推荐将类的声明与实现进行分离：

    • 头文件（.h）：存放类的声明、成员函数原型、静态变量声明等。类型的定义在 C++ 中视作一种声明，因为不直接产生机器码，也不占用运行时内存。
    • 源文件（.cpp）：存放成员函数的具体实现（函数体）。
    • 头文件使用规则：在每一个使用该类或该类函数的文件、以及编写成员函数实现的地方，都必须 #include 该头文件。

    #include 的查找逻辑

    • #include "file.h"：优先在当前 .cpp 所在的源目录搜索，未找到再去系统指定的标准库目录搜索。用于自定义头文件。
    • #include <file.h>：直接去系统指定的标准库目录（包含路径）搜索。用于标准库头文件。

    头文件防卫（Header Guards）

    为了防止同一个头文件在一个 .cpp 编译单元中被多次 #include（从而导致“类重复定义”的编译错误），必须在每个头文件中配置防卫标志：

    点击查看代码 
    #ifndef POINT_H  // 如果没有定义该宏
#define POINT_H  // 则定义该宏

class Point {
    // ...
};

#endif // 结束判断

    注：现代编译器普遍支持 #pragma once，其效果等价于上述宏防护，且编写更加简便。

    Makefile 工具与项目维护

    独立编译与链接

    • 编译单元：编译器每次只翻译一个 .cpp 文件并生成对应的目标文件（.obj 或 .o）。
    • 链接器（Linker）：将所有独立的 .obj 文件以及标准库函数“粘合”在一起，定位函数地址，最终合并生成单一的可执行文件。

    最小化编译原理

    当项目代码量巨大时，每次修改都全部重新编译将非常耗时。Makefile 是 Unix/Linux 环境下用于自动化编译的脚本工具。其核心设计目标是实现最小化编译：

    • Makefile 会将项目依赖关系构建为一张有向无环图（DAG）。

    • 时间戳对比：每次运行 make 时，工具会自动比较目标文件（Target）和依赖文件（Dependencies）的最后修改时间。

      • 如果依赖文件比目标文件更新，说明源码被改动过，则执行动作（Action）重新编译。
      • 如果目标文件比依赖文件新，说明代码未改动，则直接跳过该编译步骤。

    Makefile 基础语法与宏

    • 基本规则语法：

      点击查看代码 
      target: dependencies
    action   # 注意：action 前面必须是 Tab 键缩进！

    • 隐式规则：Makefile 默认知道 .o 文件依赖于同名的 .c/.cpp 文件，在简写时可忽略 .c 依赖。

    • 内置宏：
      • $@：代表当前规则的目标文件名（Target）。
      • $*：代表当前规则的目标文件名主干（不包含后缀部分）。

    • Clean 目标：行业惯例，定义一个没有依赖项的 clean 任务，用于清理 .o 中间件和可执行文件，恢复干净的代码环境：

      点击查看代码 
      clean:
    rm *.o sum

    • 条件语句：可以使用 ifeq、else、endif 来根据传入的变量（例如选择不同的编译器或源文件路径）控制编译的分支。
