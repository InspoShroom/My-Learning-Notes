    并发控制

    Chapter 18: Concurrency Control

    基于排队/锁的协议 (Lock-Based Protocols)

    锁是一种用于控制对数据项的并发访问的机制。

    锁的类型

    • 共享锁 (Shared Lock, S-lock): 只读锁定。若事务持有 \(D\) 的 S 锁，则它可以读取 \(D\) 但不能写入 \(D\)。多个事务可同时持有同一数据项的 S 锁。
    • 排他锁 (Exclusive Lock, X-lock): 读写锁定。若事务持有 \(D\) 的 X 锁，则它既能读也能写 \(D\)。在同一时刻，只允许一个事务持有某数据项的 X 锁。

    兼容性矩阵 (Lock Compatibility)

    请求的锁 持有的锁 S 锁 X 锁
    S 锁 ✅ 兼容 ❌ 冲突
    X 锁 ❌ 冲突 ❌ 冲突

    两阶段锁协议 (Two-Phase Locking, 2PL)

    2PL 是保证并发调度冲突可串行化的经典协议。事务执行过程分为两个阶段：

    graph LR
    Start[事务开始] --> Grow[增长阶段<br>Growing Phase]
    Grow -->|只能加锁/升级锁| Peak[顶点]
    Peak --> Shrink[收缩阶段<br>Shrinking Phase]
    Shrink -->|只能释放锁/降级锁| End[事务结束]
    • 两阶段锁变体:
    • 严格两阶段锁 (Strict 2PL): 要求事务持有的所有排他锁 (X-lock) 必须一直保持到事务结束 (Commit 或 Abort)。这保证了调度的可恢复性和无级联性。
    • 强两阶段锁 (Rigorous 2PL): 要求事务持有的所有锁 (S 锁和 X 锁) 都必须保持到事务结束。
    • 锁转换 (Lock Conversion): 增长阶段允许将共享锁升级为排他锁 (Upgrade)；收缩阶段允许将排他锁降级为共享锁 (Downgrade)。
    • 注意: 2PL 虽然能保证可串行化，但无法预防死锁 (Deadlock)。

    锁管理器与锁表 (Lock Table)

    • 锁的申请与释放由锁管理器 (Lock Manager) 处理。
    • 在内存中使用锁表 (Lock Table)（以哈希表结构实现）记录所有的加锁请求。每个数据项对应一个请求队列：
    • 队列前端深色标记表示已授予的锁，后端浅色表示等待状态的请求。

    基于图的协议 (Graph-Based Protocols)

    通过在所有数据项集合上定义一个偏序关系，将并发管理转化为有向无环图，以此作为 2PL 的替代方案。

    树状协议 (Tree Protocol)

    • 条件: 只允许使用排他锁 (X-lock)。
    • 规则:
    • 事务的首个锁可以加在任何数据项上。
    • 随后，只有当数据项 \(Q\) 的父节点已被该事务加锁时，事务才能对 \(Q\) 加锁。
    • 释放锁可以在任意时刻进行。
    • 一旦事务释放了数据项 \(Q\) 上的锁，此后就不能再次对 \(Q\) 加锁。
    • 优缺点:
    • 优点: 保证冲突可串行化；无死锁，无需回滚。
    • 缺点: 不保证可恢复和无级联；为了加锁子节点，事务可能不得不去加锁一些实际上不需要访问的祖先节点，增加了不必要的加锁开销。

    死锁处理 (Deadlock Handling)

    并发系统中，多个事务互等对方释放资源，导致所有事务均无法向前推进。

    1. 死锁预防 (Deadlock Prevention)

    • 一次性加锁 (Pre-declaration): 事务开始执行前必须一次性申请到所需的全部锁，否则不运行。
    • 偏序限制: 所有数据项按偏序加锁（如图协议）。
    • 基于时间戳抢占的预防方案:
    • 等-死方案 (Wait-Die Scheme - 非抢占):
      • 若老事务 \(T_{\text{old}}\) 请求新事务 \(T_{\text{young}}\) 占有的锁，\(T_{\text{old}}\) 等待；
      • 若新事务 \(T_{\text{young}}\) 请求老事务 \(T_{\text{old}}\) 占有的锁，\(T_{\text{young}}\) 直接回滚死亡 (Die)。
      • 口诀: “老等待，新死亡”。
    • 伤-等方案 (Wound-Wait Scheme - 抢占):
      • 若老事务 \(T_{\text{old}}\) 请求新事务 \(T_{\text{young}}\) 占有的锁，\(T_{\text{old}}\) 强行打伤并回滚 \(T_{\text{young}}\) (Wound)；
      • 若新事务 \(T_{\text{young}}\) 请求老事务 \(T_{\text{old}}\) 占有的锁，\(T_{\text{young}}\) 选择等待。
      • 口诀: “老伤害，新等待”。
    • 饥饿预防: 回滚事务重启时，必须保持其最初的时间戳，以防止老事务由于不断被新事务抢占而产生饥饿。
    • 超时机制 (Timeout): 当事务等待锁的时间超过阈值时，自动回滚。实现简单，但容易产生误判。

    2. 死锁检测与恢复 (Deadlock Detection & Recovery)

    • 等待图 (Wait-for Graph): 以活动事务为顶点。若 \(T_i\) 等待 \(T_j\) 占有的锁，则存在有向边 \(T_i \rightarrow T_j\)。
    • 死锁判定: 系统定期运行环路检测算法。等待图中存在有向环，说明发生了死锁。
    • 死锁恢复步骤:
    • 选择受害者 (Victim): 选择回滚代价最小的事务进行回滚。
    • 决定回滚范围:
      • 全回滚 (Total Rollback): 撤销整个事务并重启。
      • 部分回滚 (Partial Rollback): 仅回滚到打破死锁环路所需的最小保存点。
    • 防止饥饿: 如果同一个事务反复被选为受害者，会造成饥饿。设计时代价权重需要考虑事务的存活时间。

    多粒度锁 (Multiple Granularity)

    将数据库的数据项组织为一棵多粒度树（例如：数据库 \(\rightarrow\) 关系表 \(\rightarrow\) 数据页 \(\rightarrow\) 元组记录）。

               Database (数据库)
              /       \
         Table 1     Table 2 (关系表)
          /    \
      Page 1  Page 2 (数据页)
       /
    Record 1 (记录元组)
    • 细粒度锁 (如元组级): 并发度高，但加锁系统开销大。
    • 粗粒度锁 (如表级): 系统开销小，但并发度低。

    意向锁模式 (Intention Lock Modes)

    在对子节点显式加锁前，必须先在其所有祖先节点上加对应的意向锁，用以表明子树中存在被锁定的数据，避免粗粒度加锁时的全树扫描。

    • 意向共享锁 (Intention-Shared, IS): 表示准备在子树的更细粒度节点上加 S 锁。
    • 意向排他锁 (Intention-Exclusive, IX): 表示准备在子树的更细粒度节点上加 X 锁。
    • 共享意向排他锁 (Shared & Intention-Exclusive, SIX): 当前节点本身被 S 锁锁住，同时打算在子树的更细粒度节点上加 X 锁。

    多粒度锁兼容性矩阵

    请求 持有 IS IX S SIX X
    IS ✅ ✅ ✅ ✅ ❌
    IX ✅ ✅ ❌ ❌ ❌
    S ✅ ❌ ✅ ❌ ❌
    SIX ✅ ❌ ❌ ❌ ❌
    X ❌ ❌ ❌ ❌ ❌

    多粒度加锁规则

    1. 必须遵循上面的兼容性矩阵。
    2. 必须自顶向下（从根节点开始）进行加锁。
    3. 对节点 \(Q\) 加 S 或 IS 锁时，其父节点必须已被当前事务加上了 IS 或 IX 锁。
    4. 对节点 \(Q\) 加 X、IX 或 SIX 锁时，其父节点必须已被当前事务加上了 IX 或 SIX 锁。
    5. 解锁必须自底向上（从叶子节点到根节点）进行。
    6. 锁升级 (Lock Escalation): 当事务在某层加的细粒度锁过多时，系统会自动将其合并升级为上一层粗粒度的 S 锁或 X 锁以节省锁开销。

    幻影现象与防止 (Phantom Phenomenon)

    • 幻影现象: 当一个事务 \(T_1\) 扫描满足某范围条件的数据时，另一个并发事务 \(T_2\) 插入了满足该条件的新数据并提交。\(T_1\) 再次检索时会读取到这个新元组（幻影行）。
    • 根源: 行级锁只能锁定已存在的物理行，无法对未出现的空间进行拦截限制。

    1. 索引锁协议 (Index-Locking)

    • 规定所有查询必须通过索引完成。
    • 对索引执行 lookup 的事务，必须以 S-mode 锁定其访问过的所有索引叶子节点（即使这些节点中目前没有满足条件的记录）。
    • 执行插入/删除的事务，必须对涉及到的所有索引叶子节点加 X-mode 锁。
    • 缺点: 锁定整个叶子页会严重阻碍高并发插入。

    2. 临键锁 (Next-Key Locking)

    • 优化版本：不仅对满足索引条件的键值加锁，还要对索引顺序中的下一个键值 (Next-key value) 进行加锁。
    • 对读操作加 S 锁，写操作加 X 锁，可以完美地在保证较高并发的前提下防止幻影行的插入。

    基于时间戳排序协议 (Timestamp-Based Protocols)

    事务进入系统时被赋予唯一的时间戳 \(TS(T_i)\)。可串行化顺序由时间戳大小唯一决定。

    • 数据项 Q 上的时间戳标记:
    • \(R\text{-timestamp}(Q)\): 成功读取 \(Q\) 的所有事务中的最大时间戳。
    • \(W\text{-timestamp}(Q)\): 成功写入 \(Q\) 的所有事务中的最大时间戳。

    TSO 协议规则

    • 读操作 read(Q):
    • 若 \(TS(T_i) < W\text{-timestamp}(Q)\)：说明 \(T_i\) 试图读取一个已被后来事务覆盖的旧值。拒绝该操作，回滚并重启 \(T_i\)。
    • 若 \(TS(T_i) \ge W\text{-timestamp}(Q)\)：允许执行。将 \(R\text{-timestamp}(Q)\) 更新为 \(\max(R\text{-timestamp}(Q), TS(T_i))\)。
    • 写操作 write(Q):
    • 若 \(TS(T_i) < R\text{-timestamp}(Q)\)：说明 \(T_i\) 产生的写入值本该被前面的事务所读取，但前面的事务已经错过了。拒绝写操作，回滚 \(T_i\)。
    • 若 \(TS(T_i) < W\text{-timestamp}(Q)\)：说明 \(T_i\) 写入的是过时数据。拒绝写操作，回滚 \(T_i\)。
    • 否则：执行写操作，更新 \(W\text{-timestamp}(Q) = TS(T_i)\)。

    托马斯写规则 (Thomas' Write Rule)

    • 优化: 当 \(TS(T_i) < W\text{-timestamp}(Q)\)（即试图写入过时数据）时，直接忽略该 write 操作而不回滚事务，因为它的值反正会被更新的事务所覆盖。
    • 特点: 允许了一些非冲突可串行化但视图可串行化的调度，进一步提高了并发度。

    基于验证的协议 (Validation-Based Protocol / OCC)

    适用于读多写少、冲突极低的“乐观”场景。

    • 三阶段执行流程:
    • 读与执行阶段 (Read Phase): 事务只读取数据库，所有的写入操作都暂存在本地临时变量中。
    • 验证阶段 (Validation Phase): 事务在提交时，测试是否与其他并发事务冲突，确保满足可串行化要求。
    • 写阶段 (Write Phase): 若验证通过，将临时变量中的更新物理写入数据库；若不通过，则回滚并重新开始事务。

    快照隔离 (Snapshot Isolation, SI)

    很多现代主流关系数据库（如 Oracle, SQL Server, PostgreSQL）提供的核心并发控制机制。

    • 工作机制:
    • 事务在开始时获取一份数据库的已提交版本快照。
    • 事务的所有读操作均从快照中读取，不受并发写事务的干扰。
    • 事务写操作先写在本地，提交时遵循第一提交者赢 (First-Committer-Wins) 规则：如果并发的其他事务已经提交了对该事务想要更新的数据的修改，则该事务被迫回滚。
    • 优点: 读写互不阻塞，读性能极高，消除了脏读、丢失更新和不可重复读。
    • 写偏斜 (Write Skew) 异常:
    • SI 并不等同于完全的可串行化。 当两个并发事务基于相同的旧状态去修改不同的数据项，从而破坏全局完整性约束时，会发生写偏斜。

    eg (写偏斜): 一致性约束为 \(A + B \ge 20\)。初始 \(A=15, B=15\)。 - \(T_1\) 读取 \(A\) 和 \(B\) (快照中均为 15)，修改 \(A = A - 10 = 5\) 并提交成功。 - 并发 \(T_2\) 同时也读取 \(A\) 和 \(B\) (快照中均为 15)，修改 \(B = B - 10 = 5\) 并提交成功。 - 最终结果 \(A=5, B=5\)，打破了约束，但在快照隔离下它们修改了不同变量，无法被 SI 的写冲突检测捕获。

    • SSI (Serializable Snapshot Isolation): 快照隔离的升级版。通过在运行时追踪 Read-Write 依赖并识别可能的冲突环路，主动回滚特定事务以保证完全的 Serializable（PostgreSQL 9.1+ 采用）。

    应用层与长事务并发控制

    • 问题: 用户交互式的长事务不能长期持有底层物理锁，否则会使系统性能瘫痪。
    • 解决方案 (应用层乐观锁):
    • 表中额外加一个版本号字段 version。
    • 读出数据及版本号。

    • 写回数据时校验版本号是否改变：

      UPDATE r 
SET r.balance = r.balance + :deposit, r.version = r.version + 1
WHERE acctId = 23 AND r.version = :read_version;

    • 若 affected_rows = 0，说明在此期间该行已被其他事务修改，应用层抛出异常并提示用户重试。
