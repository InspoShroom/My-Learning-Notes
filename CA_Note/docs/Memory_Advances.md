# 第五章 存储层次结构高级篇

本章深入讨论了十种高级 Cache 优化技术、虚拟内存的安全与保护任务、虚拟机监控器（VMM）的特权要求，并结合真实硬件实例（ARM Cortex-A53 与 Intel Core i7-6700）进行了详细的地址映射与性能开销分析。

---

## 十种高级 Cache 优化技术

高级优化技术从Hit Time、Cache Bandwidth、Miss Penalty、Miss Rate、以及 Parallelism 等维度对存储系统进行全方位提升：

### 1. 降低命中时间
* **Opt #1: 小且简单的第一级 Cache（Small and Simple First-Level Caches）**：
    * 痛点：直觉上我们希望 L1 Cache 越大、相联度越高越好。但是大和复杂会导致更复杂的逻辑门电路，一级拖慢 CPU 主频。
    * 做法：采用小容量、直接映射或低相联度结构，可支持更高的处理器主频并降低功耗。直接映射下，一个虚拟地址只可能存在于 Cache 的一个固定位置，CPU 查找时不需要做多路选择，甚至可以在 Tag 比较时将数据先传给 CPU，如果 Tag 比对成功，数据立刻就能用。
* **Opt #2: 路预测（Way Prediction）**：
    * 痛点：组相联的命中率确实高，但是它每次需要将一个 Set 中的所有 Way 的 Tag 都读出来比对一遍，不仅慢还很耗电。
    * 做法：在组相联 Cache 中，为每个 Set 增加预测位来预测下一次访问命中的 Way。查询时仅需进行单次 Tag 比较，若命中则只需 1 周期；若预测失败，则在下一周期检索其余 Way。

### 2. 增加 Cache 带宽
* **Opt #3: 多 Bank 缓存与流水线访存（Multibanked Caches & Pipelined Access）**：
    * 痛点：单端口或单一 Bank 的 Cache 每次只能处理一个访存请求。当多个处理器核心并发读取数据，或者单核处理器单周期发射多条 Load/Store 指令时，单一的物理 Cache 端口会发生严重的结构冲突，成为吞吐的重大瓶颈。
    * 做法：将 Cache 划分为多个物理上独立、可并行工作的 Bank。利用顺序交叉编址（如将地址低位作为 Bank 索引）把连续的物理块映射到不同的 Bank，如果多路读写刚好打在不同的 Bank 就能并发执行。同时，将 Cache 的寻址与比对步骤进行流水线化，允许在当前请求未完成前就输入下一个请求，大幅提升吞吐率（但流水线级数加深会略微增加 Load 的命中延迟）。
* **Opt #4: 非阻塞 Cache（Nonblocking Caches）**：
    * 痛点：在乱序执行处理器中，一旦发生一次 Cache Miss，如果整个 Cache 都暂停等待数据从主存读回，那后续本来可以在 Cache 中命中的读写指令都会被无辜挂起，导致乱序执行引擎完全无法发挥隐藏访存延迟的作用。
    * 做法：采用非阻塞设计的 Cache，在发生 Miss 时不阻塞后续的内存访问。允许继续处理命中的请求（Hit under Miss）甚至其他的 Miss 请求（Miss under Miss）。硬件上利用**不命中状态寄存器（Miss Status Holding Register, MSHR）**记录每一个未完成的 Miss 物理地址、目标寄存器等状态，待主存数据返回后进行广播 and 唤醒。
    * **性能量化对比**：以 32 KiB 缓存且 L2 不命中惩罚为 10 周期为例，支持 1 个不命中下命中（Hit under 1 Miss）在 SPECInt 编译集上能带来约 9% 的性能提升，在 SPECFp 上能带来 12.5% 的提升，其效果与直接将 Cache 从直接映射升级为“2路组相联”大致相当，且非阻塞机制的硬件开销通常更具性价比。

### 3. 降低不命中开销
* **Opt #5: 关键字优先与尽早启动（Critical Word First & Early Restart）**：
    * 痛点：发生 Cache Miss 时，CPU 需要将一整个内存块（例如 64 字节）全部调入 Cache。如果 CPU 只是急需该块中的某一个字（4 字节），却必须傻傻地等待整个 64 字节块全部传输完毕才恢复执行，会产生极长的无谓等待。
    * 做法：
        * **关键字优先（Critical Word First）**：主存先发送发生 Miss 的那个特定数据字（Critical Word），一旦该字到达立即送回 CPU 并使其恢复执行，其余数据字则在后台静默填充到 Cache。
        * **尽早启动（Early Restart）**：数据依然按正常物理顺序读入，但在传输过程中，只要 CPU 请求的那个字一到达，CPU 就立即重启执行，余下数据在后台继续传输填充。
* **Opt #6: 合并写缓冲区（Merging Write Buffer）**：
    * 痛点：CPU 使用写回（Write-Back）策略时，写缓冲区（Write Buffer）负责暂存要写回主存的脏数据。如果每一次写入都直接占用总线去写主存，会产生高昂的总线抢占开销，且如果写入的数据地址离得很近，还会引发频繁的主存写冲突。
    * 做法：在写缓冲区中增加合并逻辑。当写入一个新项时，先检查缓冲区中是否已有相同块或相邻地址的写请求。如果地址连续或相邻，则将多次小位宽的写入合并为一个整块大字写操作再送往总线（例如将 4 个独立的单字写合并为一个双字或块写），显著减少主存写入次数，节省总线带宽。

### 4. 降低不命中率
* **Opt #7: 编译器优化（Compiler Optimizations）**：
    * 痛点：硬件 Cache 的容量和结构是固定的，如果程序员写出的循环访存顺序很乱，就会导致 Cache Line 刚被加载进来就被置换出去，产生大量 Capacity Miss。直接扩大硬件 Cache 成本太高。
    * 做法：在不改动硬件的前提下，利用编译器在编译期对指令和数据排布进行逻辑重构：
        * **循环嵌套交换（Loop Interchange）**：交换循环的内外嵌套层数，使程序在多维数组上的循环遍历方向（如按行还是按列）与数组在内存中的物理排布（行优先/列优先）方向完全吻合，最大化利用空间局部性。
        * **分块技术（Blocking/Tiling）**：对于矩阵乘法等大量重复读写数据的场景，将大矩阵切分成一个个尺寸极小、能完全塞入 L1 Cache 的子块（Block/Tile），在子块被置换出 Cache 之前完成全部运算，最大化挖掘时间局部性。

### 5. 并行降低不命中率与开销
* **Opt #8: 硬件预取（Hardware Prefetching）**：
    * 痛点：Cache 总是处于“被动响应”状态（发生了 Miss 才去调入），这就意味着每次首次访问都会发生 Compulsory Miss / Cold Miss。
    * 做法：在 CPU 内置独立的硬件预取器。预取器通过监测 CPU 访存流的地址步长规律，在数据被实际需要之前，提前向主存发起预取请求，将数据和指令拉入 Cache 或专属的流缓冲器（Stream Buffer）中。当 CPU 真正发起读取时，数据已在缓存中，直接实现零延迟命中。
* **Opt #9: 编译器预取（Compiler Prefetching）**：
    * 痛点：硬件预取器只能根据历史规律进行推测，对于高度复杂的非线性访存，硬件预取可能会误判导致预取了大量垃圾数据，反而挤占了有用的 Cache 行。
    * 做法：由编译器在编译期分析源码中的访存轨迹。在预计会发生 Miss 的 Load 指令前几十个周期（例如前几轮循环），在指令流中显式插入非阻塞式的**预取指令（Prefetch）**。当执行到预取指令时，数据在后台静默加载到 Cache，当真正的 Load 指令执行时直接命中，消除流水线等待开销。包括寄存器预取（Prefetch to Register）与缓存预取（Prefetch to Cache）。
    
    ??? example "编译器预取不命中次数与时钟周期计算"
        **已知条件**：
        分析以下嵌套循环（包含 3 $\times$ 100 = 300 次迭代）：
        ```c
        for (i = 0; i < 3; i = i + 1) {
            for (j = 0; j < 100; j = j + 1) {
                a[i][j] = b[j][0] + b[j+1][0];
            }
        }
        ```
        * 块大小（Block Size）为 16 字节，采用写回（Write-Back）策略。
        * 数组 `a` 和 `b` 的元素大小为 8 字节（双精度浮点数 $\implies$ 每个 Cache 块包含 $16 / 8 = 2$ 个元素）。
        * 发生 Cache Miss 的开销为 100 周期。在无 Cache Miss 时，单次迭代执行基准时间为 7 周期。
        
        **1. 无预取（No Prefetching）情况下的不命中与周期分析**：
        * **数组 `a` 不命中**：`a` 为 $3 \times 100$ 的二维数组。由于按行连续写入且每块容纳 2 个元素，因此每行产生 $100 / 2 = 50$ 次不命中。
          * `a` 总不命中数 = $3 \text{ 行} \times 50 \text{ 次/行} = 150$ 次。
        * **数组 `b` 不命中**：访问项为 `b[j][0]` 与 `b[j+1][0]`。由于是行优先存储，相邻行的第 0 列元素在内存中物理地址相差 100 个元素（800字节），它们绝对位于不同的 Cache 块中。每一次迭代对 `b[j][0]` 的访问都会发生不命中。
          * `b` 的不命中包含 `b[0][0] ~ b[100][0]`，总不命中数 = $101$ 次。
        * **总计**：$150 + 101 = 251$ 次不命中。
        * **总执行时间**：$300 \text{ 迭代} \times 7 \text{ 周期/迭代} + 251 \text{ Misses} \times 100 \text{ 周期/Miss} = 27200 \text{ 周期}$。
        
        **2. 引入编译器预取（Compiler Prefetching）优化后的分析**：
        * 编译器可以通过提前预取 7 轮迭代后的数据来消除大部分 Miss 延迟（因为 $100\text{ 周期 Miss 延迟} / 7\text{ 周期迭代} \approx 14$ 拍，考虑到预取指令自身的开销，每次迭代增加到约 8~9 周期，提前 7~8 轮预取刚好可以完全掩盖访存延迟）。
        * 优化后，仅在循环最开始的启动阶段（Startup Phase）发生无法避免的强制性不命中：
          * 数组 `a` 在每一行前 8 轮产生 $8 / 2 = 4$ 次不命中：
            * $i=0$ 行前段 Miss 数 = 4。
            * $i=1$ 行前段 Miss 数 = 4。
            * $i=2$ 行前段 Miss 数 = 4。
          * 数组 `b` 的前 7 轮发生不命中：`b[0][0] ~ b[6][0]` 共 7 次 Miss。
          * **总计不命中数**：$4 + 4 + 4 + 7 = 19$ 次。
        * **优化后总执行时间**：
          * 启动预取循环（9拍/代）：$3 \times 8 \text{ 周期} = 24 \text{ 周期}$。
          * 主循环（8拍/代，增加了预取指令开销）：$3 \times 92 \text{ 周期} = 276 \text{ 周期}$。
          * 累加 19 次 Miss 惩罚，总时间缩短为大约 **4400 周期**。
          * **结论**：预取成功避免了 232 次不命中，整体性能提升了近 6 倍。
* **Opt #10: HBM 用作 L4 Cache（HBM as L4 Cache）**：
    * 痛点：虽然利用 HBM（高带宽内存, High Bandwidth Memory）作大容量 L4 Cache 可以极大缓解访存开销，但由于 L4容量巨大（数兆至数吉字节），如果按照传统方式存储 Tag，Tag 本身就会占用极大的主存空间。而如果把 Tag 放在 HBM 内部，每次访问 L4 就必须执行两次访存（一次读 Tag，一次读 Data），这会使命中时间（Hit Time）直接翻倍。
    * 做法：
        * **同行优化（Same-row Optimization）**：利用 DRAM 读写的空间局部性，将 Tag 与其对应的 Data 强行存放在 HBM 的同一物理行（Row）中。第一次读取 Tag 时，数据所在行会缓存到 Row Buffer 中，第二次读 Data 时直接在 Row Buffer 命中，免去了耗时的行打开步骤。
        * **合金缓存（Alloy Cache）**：设计为直接映射结构，将 Tag 和 Data 打包融合成单次突发传输结构。访问 L4 时只需单次 HBM 存取即可同时获取 Tag 与 Data 进行比对，如果 Tag 比对失败（Miss），则利用特殊的存储预测器直接预判并快速重定向，从而大幅节省读取延迟。

## 虚拟内存安全保护与虚拟机

虚拟内存的核心价值之一在于为操作系统提供精细的权限隔离与安全隔离。

### 1. 体系结构保护的四大任务
为了保障多用户多进程环境的安全，硬件设计必须提供以下四个保护支撑：

1. **多特权模式**：处理器至少支持两种运行模式，即用户模式（User Mode）与系统内核模式（Supervisor/Kernel Mode）。
2. **状态保护**：提供一部分用户程序只能读取、无法写入的特权处理器状态（如页表基址寄存器）。
3. **特权级切换**：提供用户态安全进入内核态（如系统调用 `syscall`/`trap`）以及退回用户态的硬件转换机制。
4. **地址段隔离**：提供内存段隔离机制，使得在进程上下文切换（Context Switch）时，无需将整片内存置换到磁盘即可完成内存权限保护。

### 2. 虚拟机与虚拟机监控器 (VMM)
虚拟机（Virtual Machine, VM）提供了一个比完整操作系统代码库小得多的保护基（Code Base），极大地提高了数据中心的隔离性与安全性：

* **虚拟机监控器（Virtual Machine Monitor, VMM / Hypervisor）**：控制系统硬件资源，负责为客户机操作系统（Guest OS）提供与物理机几乎完全一致的运行环境，同时保持极低的主机（Host）性能开销。
* **VMM 的特权指令要求**：VMM 必须在系统特权模式下运行。客户机执行的任何试图修改系统资源（如修改页表、分配物理内存）的指令都必须是**特权指令（Privileged Instructions）**，能在用户态下触发 Trap 捕获，交由 VMM 进行模拟和托管。

## 处理器存储示例

### 1. ARM Cortex-A53 实例 (以 32 位物理地址寻址为例)
Cortex-A53 是一款支持双发射的顺序处理器，采用两级 TLB 和两级 Cache：

* **基本参数**：
    * Page Size：64 KiB ($2^{16}$ B) $\implies$ **Page Offset 为 16 位**，虚拟页号（VPN）为 16 位。
    * L1 指令 Cache：32 KiB，2路组相联，块大小 64B。
    * L1 数据 Cache：32 KiB，4路组相联，块大小 64B。
    * L2 Cache：1 MiB，16路组相联，块大小 64B。
* **地址字段位数推导**：
    * **L1 I-Cache (VIPT)**：
        * 组数 = $32\text{ KiB} / (64\text{B} \times 2) = 256$ 组 $\implies$ **Index 为 8 位**，Block Offset 为 6 位。
        * Index + Block Offset = $8 + 6 = 14$ 位。由于 $14 \le 16$（Page Offset），故 VIPT 完全不会产生别名冲突，物理 **Tag 为 18 位**（$32 - 8 - 6 = 18$）。
    * **L1 D-Cache (VIPT)**：
        * 组数 = $32\text{ KiB} / (64\text{B} \times 4) = 128$ 组 $\implies$ **Index 为 7 位**，Block Offset 为 6 位。
        * Index + Block Offset = $7 + 6 = 13$ 位 $\le 16$ 位（Page Offset），无别名冲突，物理 **Tag 为 19 位**。  
    * **L2 Cache (PIPT)**：
        * 组数 = $1\text{ MiB} / (64\text{B} \times 16) = 1024$ 组 $\implies$ **Index 为 10 位**，Block Offset 为 6 位。
        * L2 直接使用物理地址索引，其物理 **Tag 为 16 位**（$32 - 10 - 6 = 16$）。

### 2. Intel Core i7-6700 实例
Core i7-6700 是一款四核乱序执行的动态调度处理器：

* **寻址范围**：支持 48 位虚拟地址和 36 位物理地址（最大支持 36 GiB 物理内存）。
* **Cache 机制**：L1 缓存采用 VIPT 以最小化命中延迟；L2 和 L3 缓存采用 PIPT。
* **侵略性硬件预取**：为了最大程度避免长延迟导致流水线暂停，Core i7 部署了极其激进的预取策略。其 L2 的预取请求次数达到了实际 L2 需求读取次数的 1.5 倍之多。虽然这导致预取请求自身的不命中率高达 58%（并带来了可观的带宽和功耗惩罚），但成功用预取 Miss 代替了 CPU 需求访问 Miss，大幅缓解了流水线挂起。
