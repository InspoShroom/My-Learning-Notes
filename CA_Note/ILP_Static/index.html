    第六章 ILP与静态调度

    本章探讨了指令级并行（Instruction-Level Parallelism, ILP）的基本概念，多周期浮点流水线的执行延迟与冲突检测机制，分析了超流水线架构（MIPS R4000）特征，以及利用编译器进行静态调度与分支处理的高级技术。

    流水线回顾

    经典流水线总是满足以下几个基本规律：

    1. 一项任务包含一系列阶段
    2. 任务之间存在依赖关系
    3. 同时使用不同的资源来加快速度
    4. 最慢的阶段决定了完成时间
    5. 单个任务的完成时间并没有发生缩减，反而是整个任务的平均执行时间大幅缩短了
    6. 在理想状态下，即Balanced Pipeline，均衡流水线，每个阶段的持续时间都相等，k级流水线的最大加速比为k。

    多周期浮点流水线

    由于浮点（Floating-Point, FP）的运算耗时更长，无法在单个时钟周期内完成。所以我们要为浮点运算做出一些改变：

    1. 允许更长的延迟：承认浮点指令在EX阶段需要超过一个周期的时间
    2. 重复执行阶段：浮点指令默认占用多个执行周期，可以将执行单元设计成多级的子流水线来实现
    3. 使用多个浮点功能单元：比如FP Adder，FP Divider等

    延迟与重复间隔

    在流水线设计中，有两个核心性能指标：

    • 延迟（Latency）：产生运算结果的指令与使用该结果的指令之间，必须间隔的最小时钟周期数。它等于执行流水线深度减 1。ALU 的延迟时间就是0，因为其结果可以在下一时钟周期内直接使用一个核心公式是：

      \(流水线延迟 = 执行流水线深度 - 1\)

    • 重复间隔（Initiation/Repeat Interval/II）：指连续发射两个同类型操作之间必须经过的周期数。若功能单元是完全流水线化的，则重复间隔为 1

    对于典型的处理器单元，它的指标对应如下：

    执行单元 延迟（Latency） 重复间隔（Initiation/Repeat Interval） 是否流水线化
    整数 ALU 0 1 是
    主存 Load 1 1 是（有 1 周期 Load 延迟槽）
    FP 加法器 3 1 是
    FP乘法器 6 1 是
    FP 除法器 24 25 否
    常见访存与指令依赖延迟
    1. 整数 ADD \(\to\) 整数 ADD（整数ALU指令）： 
      ADD R3, R1, R2
ADD R5, R3, R4
      • Latency: 0 周期。因为 Forwarding 可直接将 EX 级输出送回 EX 级输入。
      • Initiation Interval: 1 周期。后一条指令在下一周期紧随发射。
    2. Load \(\to\) 整数 ADD（Load-Use冲突）： 
      Load R2, 0(R1)
ADD  R3, R2, R1
      • Latency: 1 周期。因为 Load 数据在 MEM 级结束才就绪，而 ADD 在 EX 级就需要操作数，必须插入 1 周期 Stalls。
      • Initiation Interval: 1 周期。
    3. Load \(\to\) Load（独立同类型指令）： 
      Load R2, 0(R1)
Load R3, 0(R1)
      • Initiation Interval: 1 周期。由于 Load 单元是完全流水线化的，第二条 Load 可以紧接着发射。
    4. Load \(\to\) Load（相关同类型指令）： 
      Load R2, 0(R1)
Load R3, 0(R2)
      • Latency: 1 周期。第二条 Load 必须等第一条 Load 的值 R2 读出后才能计算有效地址。停顿是因为 hazard，而不是访存单元忙碌。
      • Initiation Interval: 1 周期。

    冒险处理

    • Structural Hazard：
      • 成因：
        1. 非流水线化部键：例如浮点除法器通常不是流水线化的，这样同一时间就只能有一条除法指令在里面。
        2. 写端口竞争：因为不同指令（整数、加法、乘法、除法）的运行周期不同，可能会出现多条指令在同一个时钟周期试图写回寄存器堆的情况。
      • 解决方法——互锁机制：Interlock 联锁检测的两种硬件实现：
        • 译码阶段检测（Method 1）：使用一个移位寄存器（Shift Register）记录未来各个周期已被占用的写端口。新指令在译码阶段查询该移位寄存器，若预测到写回周期冲突，则在 ID 阶段暂停发射。
        • 写回阶段检测（Method 2）：允许指令发射，但在指令尝试进入 MEM/WB 写回冲突时进行暂停。当发生冲突时，优先放行执行延迟较长、流水级更深的指令（因为长延迟指令已经在流水线中消耗了更多硬件资源，将其挂起的代价远比挂起一条刚发射的短指令高）。
    • Data Hazard：
      • RAW：写后读，真数据依赖。由于浮点操作的执行时间显著增长，因等待写回导致的数据冲突与流水线暂停会大幅增加。
      • WAW：写后写，要保证寄存器最终的结果是最新的。短延迟指令在长延迟指令之后发射，但可能比其提前写回同一个寄存器，从而发生覆写错误。
        • 解决办法 1：检测到与在执行指令有相同的目标寄存器，则在 ID 级延缓后续短延迟指令的发射。
        • 解决办法 2：在长延迟指令到达写回时，由硬件关闭其写控制信号直接丢弃其已失效的过时写回结果，使后发射的短指令结果生效。
      • WAR：读后写，在顺序发射的流水线中不会发生该冲突。

    ID阶段进行冲突检测

    在指令进入 EX 之前，必须在ID级由中心化逻辑进行以下三项危险检测：

    1. Structural Hazard：检查目标功能单元是否空闲；并通过写端口移位寄存器，确保该指令需要 WB 的那个时钟周期写端口未被预占。
    2. RAW：检查当前指令的源寄存器是否与流水线中任何已发射但未写回的目标寄存器重名。若有且无法前递，则暂停。
    3. WAW：检查是否有任何已在 A1–A4、D、M1–M7 中执行的指令，其目标寄存器与当前 ID 级的指令的目标寄存器相同。若有，则在 ID 级暂停发射。

    乱序写回中的异常处理

    由于浮点指令的乱序写回（Out-of-Order Completion），前发射的指令（如 DIV）在后发射的指令（如 ADD）发生异常时可能尚未完成，这破坏了精准异常（Precise Exception）。硬件有四种解决策略：

    1. 忽略问题（Ignore）：不保证精准异常，异常发生时软硬件状态不一致。简单但难以调试。
    2. 结果缓冲（Buffer results）：将运算结果进行缓存，直到所有在它之前发射的指令都顺利执行完毕，才按顺序写回寄存器（即顺序提交，如 ROB 机制）。
    3. 历史日志/未来文件（History/Future file）：通过历史寄存器记录修改前的旧值，在异常发生时回滚到精准状态。
    4. 发射前安全判定（Issue checking）：只有确定在当前指令之前的所有指令都绝不会发生异常时，才允许当前指令发射（例如必须等待除法指令安全越过异常易发期）。

    超流水线架构：MIPS R4000 示例

    MIPS R4000 将经典的 5 阶段流水线细分为 8 阶段的超流水线（Superpipelining）结构，通过切细流水段来拉高处理器工作频率（因为 Cache 访问是时钟频率的关键瓶颈，超流水线通过分解 Cache 访存来提升主频）：

    • 八个流水级的物理动作：
      1. IF（Instruction Fetch 1）：取指第一半。选择 PC，启动指令 Cache（I-Cache）访问。
      2. IS（Instruction Fetch 2）：取指第二半。完成指令 Cache 访问并读出指令。
      3. RF（Register Fetch）：译码与读寄存器。进行译码、读取寄存器堆，并执行 ID 级的中心化 Hazard 检查，同时进行 I-Cache 的 Tag 命中检测。
      4. EX（Execution）：执行级。进行 ALU 整数运算、浮点地址计算，以及分支指令的目标地址计算与条件判定。
      5. DF（Data Fetch 1）：数据访存第一半。启动数据 Cache（D-Cache）访问。
      6. DS（Data Fetch 2）：数据访存第二半。完成 D-Cache 访问，数据读出就绪。
      7. TC（Tag Check）：标签比对。比对 D-Cache 的 Tag，判断数据访问是否命中（Hit）。
      8. WB（Write Back）：写回级。将 Load 或寄存器-寄存器操作的值写回目标寄存器。
    • 八级流水线下的 Hazards 惩罚分析： 在深流水线中常常会见到这两种延迟
      • Load Latency：加载延迟。因为数据直到 DS 级末尾才真正从 Cache 读出就绪。EX 级需要使用 Load 数据的后续指令，必须从 DS 级末尾或 TC 级前推，产生 2 周期空载延迟（Load Stall）。
      • Branch Delay：分支延迟。采用 Predicted-Not-Taken 静态策略。由于分支的目标 PC 计算和条件判定都在 EX 级完成：
        • 若分支实际 Taken（跳转）：指令已经在流水线中流过了 IF, IS, RF 三级，这三条投机获取的指令必须被冲刷（Flush）成 no-op。产生 3 周期分支惩罚。
        • 若分支实际 Untaken（不跳转）：下一条顺序指令已经在流水级中流过，仅有极短暂重打或不产生额外清除。产生 1 周期分支惩*。

    ILP——Instruction-Level Parallelism

    依赖关系

    Dependence

    指令之间的依赖关系决定了指令能否并行执行。下面我们始终假定指令 i 在指令 j 之前发射。

    1. 数据依赖（Data Dependence）：当指令 j 需要读取指令 i 的计算结果时，发生数据依赖。同时，数据依赖是有传递关系的，所以多条指令之间可能有一条依赖链。这是真依赖，会引起 RAW 冲突。
    2. 名称依赖（Name Dependence）：两条指令使用相同的寄存器名，但彼此没有真正的数据流动。可再细分为两类：
      • 反依赖（Antidependence）：指令 j 写，指令 i 读（易引起 WAR 冲突）。
      • 输出依赖（Output Dependence）：指令 i 和指令 j 均向同一个寄存器写入（易引起 WAW 冲突）。
      • 消除方法：寄存器重命名（Register Renaming），重新分配无关的空闲寄存器名以解除冲突。
    3. 控制依赖（Control Dependence）：指令的执行受制于分支跳转指令。
    4. 指令重排约束（Property Constraints）：编译器在重排指令时，必须遵守以下两条物理底线，不得随意调动指令：

      • 异常行为约束（Exception Behavior）：指令执行顺序的任何改变，都绝不能引入程序原本不会触发的新异常。

        异常行为约束违背实例（内存保护异常） 
        add x2, x3, x4
beq x2, x0, L1    ; 分支指令
ld  x1, 0(x2)     ; 读内存指令，控制依赖于 beq
L1: ...
        • 分析：如果 x2 包含的地址是非法地址（如空指针），原本程序在 beq 跳转时会直接绕过 ld，不引发任何异常。但如果编译器强行将 ld 指令重排（提升）到分支 beq 之前以提前加载数据，就会无中生有地触发一个内存访问保护异常（Page Fault），导致程序非法崩溃。
          • 数据流约束（Data Flow）：分支指令会使得同一个寄存器的来源变得动态且不确定，重排必须确保数据从“生产者”指令正确流向“消费者”指令，不能改变其依赖的动态流向。 ???+ example "数据流约束违背实例（动态数据流向）" 
            add x1, x2, x3
beq x4, x0, L
sub x1, x5, x6
L: ...
or  x7, x1, x8    ; x1 依赖于 beq 分支是否跳转
        • 分析：or 指令所消费的 x1 的值，取决于分支 beq 是否发生（可能来自 add，也可能来自 sub）。如果把 or 提到 beq 之前，就会破坏这一动态数据流路径，导致 or 读取到错误的 x1 版本。

    静态调度与循环展开

    Scheduling & Loop Unrolling

    静态调度经典示例

    对于以下 RISC-V 循环体： for (i = 999; i >= 0; i--) x[i] = x[i] + s;

    • 版本一：无调度时 

      Loop: fld f0, 0(x1)
      ; [Stall 1 CC] 等待 fld 结果写入 f0
      fadd.d f4, f0, f2
      ; [Stall 2 CC] 等待 fadd.d 结果写入 f4
      fsd f4, 0(x1)
      addi x1, x1, -8
      bne x1, x2, Loop  ; 若未对齐，addi 到 bne 可能还需暂停

      • 耗时：5 条指令 + 3 周期暂停 = 8 周期/次。

    • 版本二：流水线静态调度优化 利用无关指令 addi 填补 fld 的延迟槽，并调整 fsd 的寻址偏移量： 

      Loop: fld f0, 0(x1)
      addi x1, x1, -8
      fadd.d f4, f0, f2
      ; [Stall 2 CC] 等待 f4 写回
      fsd f4, 8(x1)     ; 偏移量调整为 8
      bne x1, x2, Loop

      • 耗时：5 条指令 + 2 周期暂停 = 7 周期/次（实际做功指令仅 3 条，其余为循环开销）。

    • 版本三：仅循环展开但未调度 仅将循环体展开 4 次并进行寄存器重命名（避免 WAW/WAR 冲突），但保持原指令顺序，不进行重排： 

      Loop: fld f0, 0(x1)
      ; [Stall 1 CC] 等待 fld
      fadd.d f4, f0, f2
      ; [Stall 2 CC] 等待 fadd.d
      fsd f4, 0(x1)

      fld f6, -8(x1)
      ; [Stall 1 CC]
      fadd.d f8, f6, f2
      ; [Stall 2 CC]
      fsd f8, -8(x1)

      fld f10, -16(x1)
      ; [Stall 1 CC]
      fadd.d f12, f10, f2
      ; [Stall 2 CC]
      fsd f12, -16(x1)

      fld f14, -24(x1)
      ; [Stall 1 CC]
      fadd.d f16, f14, f2
      ; [Stall 2 CC]
      fsd f16, -24(x1)

      addi x1, x1, -32
      bne x1, x2, Loop

      • 耗时分析：包含 14 条指令。每次迭代仍有 3 周期 Stall（1 周期 Load 延迟 + 2 周期 FP Add 延迟），共 \(4 \times 3 = 12\) 周期 Stall。总周期数 = 14 条指令 + 12 周期 Stall = 26 周期/4次迭代，均摊每次迭代 6.5 周期（仅通过平摊循环控制开销获得了微小提升）。

    • 版本四：循环展开并进行静态调度 为了获得最大性能，编译器在展开 4 次的基础上进行指令重排（Scheduling），利用不相关的指令交错填充所有延迟槽： 

      Loop: fld f0, 0(x1)
      fld f6, -8(x1)      ; 寄存器重命名 f0 -> f6
      fld f10, -16(x1)    ; 寄存器重命名 f0 -> f10
      fld f14, -24(x1)    ; 寄存器重命名 f0 -> f14
      fadd.d f4, f0, f2
      fadd.d f8, f6, f2   ; 寄存器重命名 f4 -> f8
      fadd.d f12, f10, f2 ; 寄存器重命名 f4 -> f12
      fadd.d f16, f14, f2 ; 寄存器重命名 f4 -> f16
      fsd f4, 0(x1)
      fsd f8, -8(x1)
      fsd f12, -16(x1)
      addi x1, x1, -32    ; 一次性递减指针
      fsd f16, 8(x1)      ; 调整最后一次写入的偏移量
      bne x1, x2, Loop

      • 耗时：所有指令延迟槽均被无关的并行操作填充，没有发生任何暂停。14 条指令共耗时 14 周期，均摊每次迭代仅需 3.5 周期/次。

    迹调度与超块

    对于存在大量分支跳转的非结构化代码，编译器可以通过全局指令重排进行优化：

    • 迹调度（Trace Scheduling）：识别程序中发生概率极高（如 99%）的一条热路径（称为 Trace），并将其视为无分支的长基本块进行重排。包含三个核心步骤：
      1. 迹选择（Trace Selection）：通过静态预测或运行时 Profile 找出发生频率最高的长基本块序列（热路径）。
      2. 迹压缩（Trace Compaction）：在假设分支不发生的情况下，对热路径上的指令进行跨块重排与并行压缩，填入 VLIW 的指令槽中以提高吞吐。
      3. 修复代码（Fixup / Compensation Code）：在热路径中分支跳转移出（Side-exit）或流回（Side-entrance）的接口处，插入特殊的补偿代码，用以确保当代码流偏离热路径时的逻辑与寄存器状态正确。
    • 超块（Superblock）：
      • 起因：由于迹调度在热路径中保留了多个侧入口（Side-entrance），使得生成修复代码和跟踪数据流的复杂度极高。
      • 解决（尾复制 Tail Duplication）：超块通过复制侧入口下游的所有指令块，并把侧入口的跳转点重定向到这些克隆出来的块上，从而消除了超块内部的所有侧入口。超块转变为“单入口、多出口（Single-entry, Multiple-exits）”的网格结构，极大地简化了编译器做全局调度和去除公共子表达式的难度。

    分支冲突静态解决方法

    四种静态分支策略

    1. 流水线冻结/冲刷（Freeze/Flush）：分支指令译码后，冻结所有后继指令获取，直到跳转目标确定。产生固定开销。
    2. 预测不跳转（Predicted Not-Taken）：默认继续获取 PC+4 的指令。若实际跳转，则将流水线中已获取的指令冲刷为 no-op，并从目标地址重新取指。
    3. 预测跳转（Predicted Taken，Slide 133-134）：默认分支跳转发生并从目标地址取指。
      • 适用限制：该策略在经典 5 级流水线中无效（因为分支在 ID 级完成，判定结果与目标地址同时得出，无提前预测空间）。它仅适用于分支目标地址比判定结果更早获知的流水线（例如条件在 EX 级用 ALU 计算，但目标地址在 ID 级已算出，此时可提前 1 周期按预测的 Target 取指）。

    4. 延迟分支（Delayed Branch）：分支指令后紧随的一条指令空间称为分支延迟槽（Branch Delay Slot）。无论分支实际跳转与否，延迟槽中的指令必定会被执行，编译器常用与分支无关的指令填充该槽来消除暂停。

      分支延迟槽的三种静态填充策略

      编译器填充延迟槽时有以下三种典型策略：

      • 策略 A：从分支前填充（From Before）

        • 做法：选择一条原本就在分支指令之前的、与分支判断无关的独立指令，将其移动到延迟槽中。
        • 适用性：最安全、最推荐。无论分支是否发生跳转，被移动的指令都必须执行，没有任何副作用。

      • 策略 B：从跳转目标处填充（From Target）

        • 做法：从分支跳转的目标基本块（Target Block）头部复制一条指令填入延迟槽，并将分支跳转目标修正为该指令的下一条。
        基本块

        它是一段小规模的代码序列，既满足没有分支进入到它这里（除非它是入口），也保证没有分支从它这里出去（除非它是出口）。

        • 适用性：适合分支跳转概率高（Taken）的场景。如果分支最终未跳转，延迟槽中的指令必须对后续无影响（或硬件支持在未跳转时取消该指令的写回）。

      • 策略 C：从顺序后继处填充（From Fall-Through）

        • 做法：从分支不跳转时的顺序后继基本块（Fall-Through Block）头部复制一条指令填入延迟槽。
        • 适用性：适合分支跳转概率低（Not-Taken）的场景。如果分支最终发生了跳转，延迟槽中的指令必须对跳转目标路径无副作用。

    深流水线下分支预测对性能的制约

    • 条件：取指级深度为 20 周期（即预测错误惩罚为 20 周期），程序中平均每 5 条指令含 1 个分支。
    • 不同预测准确率下的 500 条指令取指周期计算：
      • 100% 准确度：不需要任何惩罚周期，耗时 = \(500 \text{ 周期}\)。
      • 99% 准确度：500条指令包含 100 个分支，预测错误 1 次。耗时 = \(500 + 20 \times 1 = 520 \text{ 周期}\)。
      • 90% 准确度：预测错误 10 次。耗时 = \(500 + 20 \times 10 = 700 \text{ 周期}\)（性能退化达 40%）。
    • 结论：流水线越深，分支预测准确率的微小提升对性能的增益就越关键。
