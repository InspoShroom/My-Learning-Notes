    第九章 数据级并行与矢量/GPU 架构

    本章介绍数据级并行（Data-Level Parallelism, DLP）的基本原理，重点剖析向量处理器架构（RV64V）、车队时间模型、内存多 Bank 设计与步长冲突、多媒体 SIMD 架构（以及屋顶图模型），以及 GPU 架构（SIMT 编程模型、分支分歧与同步栈、内存分层）的运行机制。

    数据级并行与向量架构

    1. 什么是数据级并行（DLP）

    • 基本定义：数据级并行是指对大规模数据集合中的不同数据元素，同时执行相同的操作。
    • 为什么能效更高：在传统的指令级并行（ILP）中，每个操作都需要经过取指、译码等控制步骤；而 DLP 通过“单条指令启动大量独立数据操作”，大幅削减了取指、译码和依赖检查的硬件能量消耗，能效极高。
    • 对程序员更友好：相比于经典的 MIMD（多指令多数据）多线程并发编程，DLP 不需要程序员去处理繁琐的线程同步、锁和数据竞争问题。

    2. 向量架构基本概念

    • 工作原理：
      1. 从内存中抓取分散的数据元素（Gather）。
      2. 将它们成组放置在顺序的向量寄存器文件中（Vector Register File）。
      3. 用流水线化的功能单元对这些寄存器数据执行计算。
      4. 将计算后的向量结果分散写回内存中（Scatter）。
    • 优势：一条向量操作指令可以触发几十次独立的寄存器-寄存器操作，且各数据元素相互独立，天然无数据冒险。

    3. RV64V 向量处理器设计要点

    • 向量寄存器堆：包含 32 个向量寄存器。它们通过高读写端口数的交叉开关（Crossbar Switches）与向量功能单元相连。
    • 完全流水线化：每个向量单元都是完全流水线化的，每个时钟周期都能接收并开始处理一个新元素的操作。

    • 指令格式后缀：向量指令多使用 R 类型指令，支持三类后缀：

      • .vv：操作数均为向量寄存器（Vector-Vector）。
      • .vs：第一操作数为向量，第二操作数为标量寄存器（Vector-Scalar）。
      • .sv：第一操作数为标量，第二操作数为向量寄存器（Scalar-Vector）。

    动态寄存器类型与空间重分配

    • 动态寄存器类型（Dynamic Register Typing）：
      • 为什么：如果在指令编码中为每一种数据类型（8位、16位、32位、64位整型/浮点）都分配独立的指令码，会导致指令集过度膨胀。因此，RV64V 的指令编码中省略了数据类型和大小的定义。
      • 怎么做：程序在执行向量操作前，先通过系统指令配置寄存器的数据宽度与类型（整型：8/16/32/64位；浮点：16/32/64位）。
    • 最大向量长度（MVL）的动态调整：
      • 机制：允许程序禁用不需要的向量寄存器，从而将空闲出的物理存储空间分配给启用的寄存器，成倍增加其最大向量长度（Maximum Vector Length, MVL）。
      • 算例：假设向量内存总容量为 1024 字节：
      • 若启用全部 32 个向量寄存器，每个寄存器分配到 32 字节容量。
      • 若程序配置仅启用 4 个向量寄存器，则每个寄存器可分配到 256 字节容量，MVL 随之扩大 8 倍。
    • 隐式类型转换：支持根据寄存器当前的位宽配置直接在硬件中进行隐式的数据类型转换，无需额外显式插入类型转换指令。

    车队与 Chime 执行时间模型

    核心概念

    • 车队（Convoy）：可以在流水线中潜在并行执行的一组向量指令集合。我们可以把每一条向量指令看作一辆大卡车，流水线就是高速公路。
      • 结构冲突分流：若两条向量指令需要争抢同一个硬件功能单元（如两条访存指令 vld / vld 或 vld / vst），则它们必须序列化，拆分到不同的车队中执行。
      • 真相关放行：同一个车队中允许存在真数据相关（RAW）指令。因为硬件支持链接机制，第二条指令可以紧随第一条指令流水化运行。
    • 链接（Chaining）与弹性链接（Flexible Chaining）：
      • 链接类似于标量 CPU 中的前递（Forwarding）技术。前一条向量指令的单个元素刚在 FUs 中计算完毕，无需等待整条向量指令全部执行完或写回寄存器，直接通过旁路送入下一条功能单元开始执行。
      • 弹性链接允许一个向量指令链接到任意其他活跃的向量指令。早期的链接非常死板，只能在特定的时钟周期或特定的部件之间链接。弹性链接 允许任意两个活跃的向量指令在任意时刻、任意空闲的功能单元之间建立这种“直通管道”，极大地提高了车队的组装成功率。
    • 鸣响（Chime）：执行一个车队所需的时间单位。对于向量长度为 \(n\)、包含 \(m\) 个车队的序列，忽略流水线启动开销时，总执行时间约为 \(m \times n\) 个时钟周期。

    车队划分与 Cycles/FLOP（FLoating-point Operation，浮点运算） 性能算例

    假设处理器各向量功能单元只有单个备份，即只有一个端口，真相关均可链接并行。分析以下 RV64V 向量序列：

    1. vld     v0, 0(x5)    ; 向量加载 1
2. vmul.vv v4, v0, v2   ; 向量乘法（依赖 v0）
3. vld     v2, 0(x6)    ; 向量加载 2（与第一条 vld 产生访存结构冲突）
4. vadd.vv v8, v2, v6   ; 向量加法（依赖 v2）
5. vst     v8, 0(x6)    ; 向量存储 （与加载 vld 产生访存结构冲突）
    • 划分 Convoys（车队）步骤：
      1. 指令 1 (vld) 发射，指令 2 (vmul) 依赖指令 1 的结果，通过 Chaining 链接，两者可以在同一个车队中并行 \(\to\) Convoy 1。
      2. 指令 3 (vld) 也是访存操作，与前面的 vld 争抢单一的访存功能单元，产生结构冲突，必须切断，另起车队。指令 4 (vadd) 依赖指令 3，Chaining 链接，归入同一个车队 \(\to\) Convoy 2。
      3. 指令 5 (vst) 是存储操作，与 Convoy 2 中的加载 vld 发生结构冲突，必须再次另起车队 \(\to\) Convoy 3。
    • Chime 与执行时间计算：
      • 该序列被划分为 3 个 Convoys，因此需要 3 个 Chimes 来完成。
      • 若向量长度 \(n = 32\)，则总时钟周期数约为 \(3 \times 32 = 96\) 周期。
      • 运算总功：浮点加法和浮点乘法指令各处理 32 个元素，共执行了 \(32 \times 2 = 64\) 次浮点操作（FLOPs）。
      • 性能指标（每个浮点操作消耗的时钟周期数，Cycles per FLOP）： \(\text{Cycles per FLOP} = \frac{96 \text{ 周期}}{64 \text{ FLOPs}} = 1.5 \text{ 周期/FLOP}\)

    分条挖掘与向量长度控制

    • 为什么需要分条挖掘：

      硬件中向量寄存器的最大向量长度（MVL）是有限且固定的（例如 64 或 128）。当程序中需要处理的循环上限 \(n\) 大于 MVL，或者 \(n\) 在编译期无法预测时，必须在软件层进行切分。

      比如说 MVL = 64，但是用户输入了200，那么就得分批打包处理，一次处理64个元素，直到全部处理完，这个过程就是分条挖掘。

    • 分条挖掘（Strip Mining）机制：

      • 编译器会将大循环拆分为内外两层循环。
      • 外层循环处理剩下的余数 \(n \pmod{\text{MVL}}\) 个元素，将向量寄存器填满一部分。
      • 内层循环以 MVL 为固定步长成组地处理剩下的元素。
      • 在循环体中，使用 setvl 指令动态设置向量长度寄存器 vl（Vector Length）。vl 寄存器将控制后续所有向量指令实际处理的元素个数，最大不超过 MVL。

    条件控制与内存访存设计

    向量条件分支（谓词/条件掩码）

    • 为什么需要向量条件分支：

      1. 对于标量CPU，如果遇到if-else，我们就会使用如 beq, bne 等指令来让程序跳过某段代码。
      2. 对于向量处理器，它一次性处理一批数据，如果64个元素中前面30个是true后面34个是false，那么处理器就无法做出正确的判断了。
      3. 于是引入掩码机制——让所有元素都通过流水线，但我用掩码来进行过滤，只对掩码为1的元素进行计算。

    • 原理：将原来的if-else语句，转换为两步：

      1. vplt 向量比较产生掩码
      2. vadd 带掩码的向量加法
    • 掩码寄存器（Mask/Predicate Registers）：
      1. 使用 1 位宽的掩码寄存器（如 p0）记录条件判断的结果（0 或 1）。
      2. 当启用掩码寄存器控制时，后续向量操作只会对掩码对应位为 1 的元素执行计算并写入目的寄存器。
      3. 掩码为 0 的对应位置，其目的寄存器中的原始值保持不变（不作任何改写）。
    • 编译器 IF 转换：编译器通过将条件分支跳转转化为直线的谓词掩码向量操作，消除了控制冒险。

    内存多 Bank 设计与满载带宽计算

    • 为什么需要多 Bank：内存 SRAM 的忙周期（Bank Busy Time）通常是处理器时钟周期的数倍。如果单时钟周期内有大量处理器同时请求访存，且都指向同一个主存，就会产生严重的冲突导致长时间挂起。因此，内存被切分为多个独立寻址工作的物理 Bank。
    • 最小 Bank 数量计算示例：
      • 已知条件：Cray T90 系统包含 32 个处理器。每个处理器在每个时钟周期发出 4 个 Load 和 2 个 Store 请求。处理器时钟周期为 2.167 ns，主存 SRAM 忙时间为 15 ns。
      • 计算步骤：
        1. 计算系统单周期最大发出的访存请求数： $\(\text{Requests per Cycle} = (4 \text{ Load} + 2 \text{ Store}) \times 32 \text{ Processors} = 192 \text{ 次}\)$
        2. 计算以标量处理器时钟周期为单位的 SRAM 忙时间（向上取整）： $\(\text{SRAM Busy Cycles} = \frac{15 \text{ ns}}{2.167 \text{ ns}} = 6.92 \approx 7 \text{ 周期}\)$
        3. 为保证所有处理器满载运行时不发生 Bank 冲突阻塞，主存所需的最小 Bank 数量为： $\(\text{Min Banks} = 192 \text{ 请求/周期} \times 7 \text{ 忙周期} = 1344 \text{ 个}\)$

    3. 步长与 Bank 冲突判定

    • 步长（Stride）：向量中相邻两个元素在主存中的物理内存地址间隔。
      • 单元步长（Unit Stride）：步长为 1，代表要访问的元素在主存中是连续存放的。
      • 非单元步长（Non-unit Stride）：步长大于 1（例如矩阵列访问时，由于是行优先存储，相邻两个列元素地址相差一整行的数据大小）。采用 vlds / vsts 指令访存。
    • 冲突判定物理规律：当两个访问请求在 Bank 尚未空闲时映射到同一个 Bank，即产生 Bank 冲突。若步长与 Bank 数量存在公约数，会导致周期性的 Bank 命中。判定冲突导致 Stall 的充要公式为： $\(\frac{\text{LCM}(\text{Stride}, \text{Bank 数量})}{\text{Stride}} < \text{SRAM Bank 忙周期}\)$ (其中 LCM 为最小公倍数，该公式本质计算的是下一次请求回到同一个 Bank 所经过的周期数)
    • 冲突性能对比经典算例（Slide 68-72）：
      • 已知条件：系统包含 8 个 Bank，忙时间为 6 周期，总访存延迟为 12 周期，要处理一个 64 元素的向量。
      • 情况 A（Stride = 1）：
        • \(\text{LCM}(1, 8)/1 = 8 \ge 6\)。说明下一次回到 Bank 0 时已经过了 8 周期，Bank 早已恢复空闲，无任何 Bank 冲突。
        • 总消耗时间 = 总延迟 + 向量长度 = \(12 + 64 = 76\) 周期。
      • 情况 B（Stride = 32）：
        • \(\text{LCM}(32, 8)/32 = 1 < 6\)。因为 \(32 \pmod 8 = 0\)，所有 64 个元素的访存请求都会被引向同一个 Bank（Bank 0），发生极度严重的 Bank 冲突。除了第一个元素外，后续每一个元素的访问都要被迫等待一个完整的 6 周期忙时间。
        • 总消耗时间 = 总延迟 + 1（首元素）+ 6 周期 \(\times\) 63（剩余元素等待） = \(12 + 1 + 378 = 391\) 周期。性能暴跌。

    4. 收集-分散访存（Gather-Scatter）

    • 背景：在稀疏矩阵或稀疏数组运算中，矩阵中大部分元素是零，只需要处理非零元素。非零元素的物理内存地址是不规则且离散的，无法用固定步长表示，需要通过一个索引向量（Index Vector）记录偏移。

    • 工作机制：

      • Gather（收集）：用基地址加上索引向量 K 中的各个偏移值，从主存不规则的离散地址中读取数据，并在硬件中打包装入一个连续的向量寄存器中。对应指令为 vldx。
      • Scatter（分散）：计算完毕后，将向量寄存器中连续存放的结果，根据另一个索引向量 M 中给出的稀疏偏移量，分散写入到主存的不规则稀疏地址中。对应指令为 vstx。

    多媒体 SIMD 与屋顶图模型

    1. 多媒体 SIMD 架构特征

    • 对比向量架构的三大删减（高频考点）： 多媒体 SIMD（如 x86 的 AVX 指令集、ARM 的 NEON 指令集）由于面向固定长度的超短向量（如 256 位或 512 位），在硬件中省略了以下三大机制以简化设计：
      1. 省略了向量长度寄存器（vl）。
      2. 省略了非单元步长访存与收集-分散访存指令。
      3. 省略了谓词/条件控制掩码寄存器。

    2. 屋顶图性能模型（Roofline Model）

    • 基本原理：屋顶图模型将处理器的浮点吞吐性能（横轴：算术强度，单位 FLOP/Byte；纵轴：可达到的最高性能，单位 GFLOP/s）描绘在同一个二维坐标系中。

    • 核心概念：

      • 算术强度（Arithmetic Intensity）：程序运行期间每从主存读写 1 字节的数据，所执行的浮点操作次数。 $\(\text{Arithmetic Intensity} = \frac{\text{浮点操作次数 (FLOPs)}}{\text{读写内存的总字节数 (Bytes)}}\)$
      • 屋顶图分界（Roofline）：
        • 左侧斜坡（Memory-bound）：受限于内存带宽。此时算术强度低，处理器计算核心经常因为等数据而闲置，提升内存带宽或优化数据复用可提升性能。
        • 右侧平顶（Compute-bound）：受限于计算核心上限。算术强度极高，内存带宽不再是瓶颈，性能达到处理器硬件能提供的最大 GFLOP/s 上限。

    GPU 架构与 CUDA 编程机制

    GPU 是一类高度多线程的 SIMD（或称为 SIMT 单指令多线程）处理器。

    1. CUDA 线程管理与索引计算

    • CUDA 逻辑层次结构：Grid（网格） \(\to\) Thread Block（线程块） \(\to\) Thread（线程）。
      • 线程块被分配到多线程 SIMD 处理器（如 NVIDIA SM）上独立运行，线程块之间无法直接通信，且执行顺序随机。

    • 1D 线程索引计算公式（最基础的入门大题）： 当 GPU 执行并行循环时，每个物理线程必须计算出自己对应处理的数据元素下标。在 1D 网格拓扑下，计算公式为： $\(\text{index} = \text{blockIdx.x} \times \text{blockDim.x} + \text{threadIdx.x}\)$

      • blockIdx.x：当前线程块在 Grid 中的 ID。
      • blockDim.x：每个线程块中包含的线程总数。
      • threadIdx.x：当前线程在自己所在的线程块内部的线程 ID。

    • 函数装饰符：

      • __host__：在 CPU（主机）上编译且在 CPU 上运行的普通函数。
      • __device__：在 GPU 上运行，且只能由 GPU 内部函数调用的辅助函数。
      • __global__：在 CPU 上调用，在 GPU 上启动执行的核函数（Kernel）。

    2. GPU 硬件控制与二级调度器

    • 二级硬件调度：
      1. 线程块调度器（Thread Block Scheduler）：属于全局粗粒度调度，负责将整个线程块分发给空闲的多线程 SIMD 处理器（SM）。
      2. SIMD 线程调度器（SIMD Thread Scheduler）：属于 SM 内部的细粒度指令级调度，负责调度各 SIMD 线程组（Warp，包含 32 个线程）在物理 SIMD Lanes 上发射并执行。
    • PTX 虚拟指令集：PTX（Parallel Thread Execution）是 NVIDIA 引入的一层硬件无关的稳定虚拟汇编。编译器先将 CUDA 翻译为 PTX，驱动程序在程序加载运行时通过即时编译（JIT）将其翻译为对应显卡世代的物理机器码，保证了跨代兼容性。

    3. 分支分歧（Warp Divergence）与同步栈

    • 分支分歧现象：在一个 Warp（32 线程）中，如果遇到 if-else 分支，且部分 Lane 条件为真，部分为假，硬件无法同时执行两个分支。此时必须串行执行：
      • 第一步：激活满足 if 条件的线程，屏蔽（Mask 设为 0）不满足的线程，执行 THEN 分支。
      • 第二步：激活满足 else 条件的线程，屏蔽原先的线程，执行 ELSE 分支。
      • 性能衰退：在单层 if-else 下，Warp 整体执行效率会跌至 \(50\%\) 以下；嵌套两层 if-else 时执行效率跌至 \(25\%\) 以下。
    • 分支同步栈（Branch Synchronization Stack）：
      • 作用：为了在硬件指令级自动管理分支发散后的线程激活状态（Mask），硬件为每个 SIMD 线程组维护一个同步栈。
      • 栈条目结构：记录标识 Token、分支分歧跳转的目标指令地址、以及目标活跃线程掩码（Target Thread-active Mask）。
      • 运行机制：遇到分支发散时，将未执行的 ELSE 分支路径地址与对应掩码压入同步栈（Diverge），先执行当前分支；当前分支执行完毕后，硬件自动从同步栈中弹出条目并跳转到目标地址，合并不活跃掩码继续执行（Converge）。

    4. GPU 存储层次

    • Local Memory（本地内存）：线程私有。由于物理上存放在板载显存上，访问速度慢。
    • Shared Memory（共享内存）：片上高带宽低延迟的 SRAM 缓存。在同一个 Thread Block 内的所有线程共享，可用于线程间的协作与数据交换。
    • Global Memory（全局内存）：板载大容量显存。对所有线程块中的所有线程以及 CPU 宿主机完全可见，访问延迟高。
